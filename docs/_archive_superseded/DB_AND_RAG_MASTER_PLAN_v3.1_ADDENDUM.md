## 0. 이 문서의 위치

v3.0이 "무엇을 만들 것인가"(전략+스키마)를 다뤘다면,
v3.1은 "어떻게 만들 것인가"(기술 결정+데이터 설계 상세)를 다룬다.

```
v3.0 (전략 레벨) ← 유효, 수정 없음
  + v3.1 보강안 (본 문서) ← 기술 결정 + 데이터 상세 + 규범 처리 원칙
  = 구현 착수 가능한 완전한 계획
```

v3.0의 5 Phase 구조, Hybrid RAG 선택, Supabase 단일 플랫폼 결정은 그대로 유지한다.

## 2. 핵심 기술 결정 3가지

### 결정 A — 청킹 전략

**문제**: 기사 텍스트와 패턴/규범 텍스트를 어떤 단위로 임베딩할 것인가?

**결정**:

| 데이터 유형 | 청킹 전략 | 근거 |
|-------------|-----------|------|
| 패턴 소분류 | **그대로 사용** (별도 청킹 불필요) | 이미 평균 350자로 임베딩에 적합한 크기. 3단계 계층으로 자연 구조화됨 |
| 윤리규범 조항 | **세부 조항 단위** (①②... 수준) | 정확한 원문 인용을 위해 조항 단위 독립 저장 필요 |
| 기사 텍스트 (쿼리 시) | **단락 단위 임베딩 → 복수 청크 검색** | 기사 전문(1,000~3,000자)을 단일 벡터로 만들면 세부 패턴 매칭이 희석됨. 단락 단위로 분할하여 각각 검색하고, 결과를 집계 |

**기사 청킹 상세 설계**:
- 기사를 단락(paragraph) 단위로 분할 (평균 2~5개 단락)
- 각 단락을 개별 임베딩하여 패턴 DB에 대해 유사도 검색
- 검색 결과를 단락별로 수집한 뒤, 패턴별 최고 유사도를 기준으로 상위 N개 패턴 후보 도출
- 이 후보 패턴 목록 + 관계 테이블의 연결 규범을 RAG 컨텍스트로 Claude에 제공
- **기사 텍스트 자체는 DB에 저장하지 않음** (저작권 원칙 준수, v3.0과 동일)

**참조**: `01/70% of RAG Performance is Related to`, `01/15 RAG Chunking Techniques`

### 결정 B — 관계 모델링 수준

**문제**: v3.0의 관계형 테이블(3테이블)로 충분한가, 별도 온톨로지/Graph DB가 필요한가?

**결정**: **관계형 테이블로 충분. 단, 2가지를 보강한다.**

**근거**:
- 전체 엔티티 ~262개, 관계 300~500건 수준 → Neo4j 같은 전용 Graph DB는 오버엔지니어링
- Supabase 관계형 테이블의 JOIN 성능으로 이 규모의 그래프 순회는 충분히 처리 가능
- `02/When to Use a Knowledge Graph And When Its Overkill`의 결론과도 일치
- `02/The Ontology Tax`에서 지적하는 초기 구축 비용을 고려하면, 현 단계에서는 관계형이 합리적

**보강 1 — 규범 위계 구조 반영** (ethics_codes 테이블):
```
tier INT NOT NULL CHECK (tier BETWEEN 1 AND 4)
  -- 1=헌장·선언, 2=강령·요강, 3=준칙, 4=권고기준·선언

parent_code_id BIGINT REFERENCES ethics_codes(id)
  -- 같은 원칙의 상위 조항 참조 (구체→포괄 체인)

domain TEXT
  -- 적용 영역: 'general', 'suicide', 'disaster', 'election',
  -- 'human_rights', 'gender', 'disability', 'minors',
  -- 'infection', 'reunification', 'hate_speech' 등
```

**보강 2 — 패턴 간 관계 테이블 추가** (v3.0에 없던 새 테이블):
```
CREATE TABLE public.pattern_relations (
  id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  source_pattern_id BIGINT REFERENCES patterns(id) NOT NULL,
  target_pattern_id BIGINT REFERENCES patterns(id) NOT NULL,
  relation_type TEXT NOT NULL CHECK (relation_type IN (
    'co_occurs',     -- 빈번하게 동시 출현
    'escalates_to',  -- 심화 관계 (A가 있으면 B로 이어질 수 있음)
    'variant_of',    -- 같은 문제의 변형 (예: 1-7-1 ↔ 1-2-2)
    'inferred_by'    -- A+B+C 조합으로 D를 추론 (메타 패턴용)
  )),
  description TEXT,
  CONSTRAINT unique_pattern_rel UNIQUE (source_pattern_id, target_pattern_id, relation_type)
);
```

**시드 데이터**: criteria 파일의 15건+ 교차참조를 자동 파싱하여 이 테이블에 초기 삽입.
예시:
- 1-7-1(주관적 술어) → variant_of → 1-2-2(책임회피 표현)
- 1-6-1(인과관계 단순화) → variant_of → 1-3-3(인과관계 왜곡)
- 1-1-1 + 1-1-2 + 1-3-2 → inferred_by → 1-4-1(외부압력 의심)

### 결정 C — 임베딩 모델

**문제**: 한국어 성능, 비용, 차원 수 기준으로 어떤 모델을 선정할 것인가?

**결정**: **OpenAI text-embedding-3-small (1536차원)을 1차 선택, 성능 미달 시 대안 검토**

| 기준 | text-embedding-3-small | Voyage AI voyage-3 | 한국어 오픈소스 (bge-m3 등) |
|------|----------------------|--------------------|-----------------------------|
| 한국어 성능 | 양호 (다국어 지원) | 우수 (한국어 벤치마크 상위) | 양호~우수 (모델별 편차) |
| 차원 수 | 1536 (v3.0 스키마 호환) | 1024 | 1024 (모델별 상이) |
| 비용 | $0.02/1M 토큰 | $0.06/1M 토큰 | 무료 (로컬 실행) |
| API 안정성 | 매우 높음 | 높음 | 자체 인프라 필요 |
| Supabase 호환 | 바로 사용 가능 | 바로 사용 가능 | Edge Function에서 제약 |

**근거**:
- 전체 임베딩 대상이 ~262개로 극소량 → 비용 차이가 무의미 (어떤 모델이든 $0.01 미만)
- v3.0 스키마가 이미 vector(1536)으로 설계됨 → 차원 변경 시 스키마 수정 필요
- 기사 분석 시 쿼리 임베딩도 빈번하지 않음 (사용자 1회 분석당 1~5회 임베딩 생성)
- OpenAI의 한국어 성능이 충분하지 않을 경우, Voyage AI로 전환 (이때 vector 차원을 1024로 변경)
- 오픈소스는 현 단계에서 인프라 부담이 크므로 후순위

**참조**: `04/Best Open-Source Embedding Models for RAG`, `01/This New Embedding Model Cuts Vector DB`

## 3. 윤리규범 중복 처리 원칙 — "구체→포괄 롤업"

### 3.1 규범 위계 구조 (4 Tier)

```
Tier 1 — 헌장·선언 (가장 추상적, 철학적 원칙)
  └─ 언론윤리헌장 (9개 원칙)

Tier 2 — 강령·요강 (원칙을 구체화한 행동 규범)
  ├─ 기자윤리강령 (10조)
  ├─ 기자윤리실천요강 (3장 17항)
  ├─ 신문윤리강령 (7조)
  └─ 신문윤리실천요강 (16조 ~50개 세부항)

Tier 3 — 준칙·보도기준 (특정 분야의 구체적 행동 지침)
  ├─ 인권보도준칙 (9장 ~40항)
  ├─ 자살보도 윤리강령 (7조 + 실천요강)
  ├─ 자살예방 보도준칙 [추가]
  ├─ 재난보도준칙 [추가]
  ├─ 감염병보도준칙 [추가]
  ├─ 선거여론조사 보도준칙 [추가]
  ├─ 평화통일 보도 준칙 [추가]
  ├─ 군 취재·보도 기준
  └─ 생성형AI 준칙

Tier 4 — 권고기준·가이드라인·선언 (권고 수준)
  ├─ 혐오표현 반대 미디어 실천 선언 [추가]
  ├─ 성폭력 보도 세부 권고 기준
  ├─ 아동학대 보도 권고기준
  ├─ 정신건강보도 권고기준
  ├─ 장애인 보도 권고기준
  └─ 마약류 사건 보도 권고 기준
```

### 3.2 핵심 원칙: 같은 원칙의 반복은 "구체화 체인"

동일한 원칙이 여러 규범 문서에 반복 등장하는 것은 중복이 아니라,
추상도를 달리하는 **구체화 체인(concretization chain)**이다.

**DB에서의 처리**:
- 모든 조항을 독립 레코드로 유지 (삭제/병합하지 않음)
- `tier` 컬럼으로 위계 수준 표시
- `parent_code_id`로 상위 조항과 연결

**예시 — "무죄추정" 구체화 체인**:

| tier | 규범 | 조문 | parent |
|------|------|------|--------|
| 1 | 언론윤리헌장 3조 | "인권을 존중하고 피해를 최소화한다" | NULL |
| 2 | 신문윤리실천요강 7조① | "무죄로 추정된다는 점에 유의하여 명예와 인격권을 존중" | 헌장 3조 |
| 3 | 인권보도준칙 2장 2조 가항 | "수사나 재판 중인 사건을 다룰 때 단정적인 표현을 사용하지 않는다" | 실천요강 7조① |

### 3.3 리포트 인용 형식 — 하위→상위 롤업

분석 리포트에서 규범을 인용할 때, 가장 구체적인(하위 tier) 규범을 먼저 제시하고,
상위 규범으로 올라가며 원칙적 맥락을 부여한다.

```
[리포트 인용 포맷]

위반 사항: {구체적 문제 설명}

관련 규범:
① [준칙] {Tier 3/4 조항 원문} — 가장 구체적인 근거
② [강령] {Tier 2 조항 원문} — 원칙적 근거
③ [원칙] {Tier 1 조항 원문} — 포괄적 맥락
```

**RAG 검색 로직에의 반영**:
1. 벡터 검색으로 관련 패턴 후보 도출
2. pattern_ethics_relations에서 연결된 규범 조항 조회
3. 조회된 규범의 `parent_code_id` 체인을 따라 올라가며 상위 규범 수집
4. tier 역순(3→2→1)으로 정렬하여 Claude 프롬프트에 주입

### 4.1 Graph RAG 기대치 현실 조정

| 조사 문서의 주장 | 본 문서의 판단 |
|-----------------|---------------|
| "평가 정확도 +40~50% 향상" | **과대 추정.** 의료/법률 도메인 벤치마크를 그대로 투영한 수치. CR-Check은 데이터 규모가 작고(262개 엔티티), 관계도 사전 정의되므로 Hybrid RAG의 개선폭은 **정적 매핑 대비 환각 감소 + 맥락 적합성 향상** 수준으로 기대하는 것이 현실적. 정량 목표는 eval 체계 구축 후 설정. |
| "추가 비용 $0 (Context Mesh)" | **정확.** 임베딩 대상이 ~262개이므로 임베딩 비용은 사실상 $0. 런타임 비용도 쿼리당 pgvector 검색 1회 + JOIN 1회로 무시할 수준. |
| "Graph RAG 필수" | **현 단계에서는 Hybrid RAG(Vector + 관계형 테이블)로 충분.** 전용 Graph DB(Neo4j)는 데이터가 수천~수만 노드로 성장한 뒤에 재검토. v3.0의 결정을 유지. |
| "4-6주 점진적 도입" | **현실적.** 데이터 실사 결과 규모가 작으므로, Phase 0(RAG 지식베이스)는 v3.0 추정(3-5일)보다 빠르게 완료 가능. |

### 4.2 Crawl 단계 범위 확정

**충돌**: 조사 문서마다 "Crawl"에 포함되는 범위가 달랐음.
- 문서 A: Crawl = pgvector RAG + 결과 아카이빙
- 문서 B: Crawl = 결과 아카이빙만, RAG는 Phase 1.5

**확정**: v3.0의 **옵션 C(Phase 0 + Phase 1 동시 진행)**를 채택.

- Supabase 프로젝트 생성 시 RAG 테이블(patterns, ethics_codes, pattern_ethics_relations, pattern_relations) + 기본 테이블(articles, analysis_results)을 **한 번에 생성**
- RAG 검색 로직은 Phase 1(아카이빙) 배포 후 점진적으로 연결
- 이것이 가능한 이유: 데이터 규모가 작아서 스키마 전체 생성의 부담이 없음

### 4.3 윤리규범 정확 인용 원칙 실현 방법

**충돌**: 0번 입장 문서는 "1획도 틀리지 않는 정확한 인용"을 강조했지만,
이후 조사 문서들은 "유사도 검색"에 집중하며 정확 인용 보장 설계가 빠짐.

**확정**: **벡터 검색은 "후보 선정"에만 사용하고, 인용 원문은 DB에서 직접 조회.**

```
[인용 정확성 보장 흐름]

1. 벡터 검색 → 관련 패턴 후보 도출 (유사도 기반, 근사적)
2. 패턴 후보 → pattern_ethics_relations JOIN → 규범 조항 ID 도출 (정확한 관계)
3. 규범 ID → ethics_codes 테이블에서 full_text 직접 조회 (원문 그대로)
4. 조회된 원문을 Claude 프롬프트에 주입 → 리포트에 그대로 인용

핵심: 벡터 검색이 "대략 이 근처"를 찾고,
      관계 테이블이 "정확히 이것"을 지정하고,
      원문 테이블이 "글자 그대로"를 보장한다.
```

이 3단계 흐름이 Hybrid RAG의 핵심 가치이며,
벡터 검색 단독으로는 불가능한 정확성을 관계 테이블 + 원문 직접 조회로 보완한다.

## 6. RAG 컨텍스트 → Claude 프롬프트 주입 설계

### 6.1 2단계 파이프라인에서의 RAG 호출 지점

**결정**: 1단계(Haiku)에서 1회 RAG 조회, 결과를 2단계(Sonnet)에 전달.

```
기사 URL → 스크래핑 → 단락 분할
    ↓
단락별 임베딩 생성
    ↓
search_patterns_with_ethics() — [1회 RAG 호출]
(pgvector 유사도 → pattern_ethics_relations JOIN → parent chain 수집)
    ↓
RAG 컨텍스트 생성 (패턴 후보 + 규범 원문 + 위계 체인)
    ↓
Claude Haiku ← RAG 컨텍스트 주입 (1단계: 패턴 식별)
    ↓
Haiku 결과 (식별된 패턴 목록)
    ↓
Claude Sonnet ← Haiku 결과 + RAG 컨텍스트 (2단계: 상세 보고서)
    ↓
분석 리포트 생성 → Supabase 저장
```

**근거**: 1회 RAG 호출 선택 이유
- 비용 효율: 2회 호출 대비 임베딩/검색 비용 50% 절감
- Haiku의 역할은 "후보 좁히기"이므로, RAG가 제공한 전체 컨텍스트에서 패턴을 선별
- Sonnet은 Haiku가 선별한 패턴에 집중하여 상세 보고서를 작성하므로, 동일 RAG 컨텍스트를 재활용 가능
- 1단계 오탐 전파 리스크는 Haiku의 threshold 조정으로 관리 (recall 우선, precision은 Sonnet 단계에서 보완)

#### 작업 B: 규범 parent-child 매핑

**목표**: ~160개 규범 조항 각각에 tier(1~4)와 parent_code_id를 부여하여, 구체→포괄 롤업 인용이 가능하도록 관계 매핑을 완성

**절차**:
1. `Code of Ethics for the Press.md`의 모든 조항을 목록화 (ID, 규범명, 조문 번호, 조문 제목)
2. 각 조항에 tier 부여:
   - Tier 1: 언론윤리헌장의 9개 원칙
   - Tier 2: 기자윤리강령, 기자윤리실천요강, 신문윤리강령, 신문윤리실천요강의 조항
   - Tier 3: 인권보도준칙, 자살보도 윤리강령, 자살예방 보도준칙, 재난보도준칙, 감염병보도준칙, 선거여론조사 보도준칙, 평화통일 보도 준칙, 군 취재·보도 기준의 조항
   - Tier 4: 혐오표현 반대 선언, 성폭력/아동학대/정신건강/장애인 보도 권고기준의 조항
3. 주제별 구체화 체인 매핑: 각 Tier 3/4 조항에 대해, 같은 원칙을 다루는 Tier 2 조항(parent)을 식별하고, 그 Tier 2 조항의 Tier 1 parent도 연결
4. 매핑 결과를 별도 스프레드시트 또는 JSON으로 정리 → DB 시드 데이터로 활용

**매핑 기준**:
- 같은 "원칙"을 다루는가? (예: 무죄추정, 자살보도 주의, 차별금지 등)
- 하위 조항이 상위 조항의 구체적 실행 지침인가?
- 의심스러운 경우, 하위 조항의 서문/목적 조항에서 상위 규범을 참조하는지 확인

**산출물**: `ethics_codes_mapping.json` (또는 .csv) — 조항별 id, code, title, tier, parent_code_id, domain

**예상 작업량**: 중간 (LLM 보조로 초안 생성 후 수동 검수, 1~2시간)
