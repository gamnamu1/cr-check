# [감리 리포트] M5.5 — 배포 전 종합 감리(Antigravity)

### 📋 [M5.5] CR-Check 종합 독립 감리 리포트 (Antigravity)

1차 및 3차 감리자가 주목하지 못했을 **"코드베이스 깊숙한 곳의 논리적 모순과 시스템 병목"**을 파헤치는 데 초점을 맞추었습니다. 전체 파이프라인의 뼈대는 훌륭하나, M4(2-Call)에서 M5(1-Call Solo)로 급진적으로 전환하는 과정에서 발생한 설계적 파편들이 곳곳에 숨어 있습니다. 특히 **프롬프트와 DB 쿼리 조건 간의 충돌(CRITICAL)**이 발견되었습니다.

---

### [축 1] 수직 정합성 (DB ↔ 코드 ↔ 프롬프트)

**판정: ❌ FAIL (조건부)**

수직 통합의 핵심인 필드 매핑과 RPC 쿼리는 매우 정교하게 잘 맞물려 있습니다. 다만, 프롬프트 내부 논리에서 치명적인 모순을 발견했습니다.

1. **[CRITICAL] Few-shot 예시와 DB 공급 카탈로그 간의 모순 (패턴 1-4-2)**
    - **원인**: `pattern_matcher.py`의 `_SONNET_SOLO_PROMPT`에는 `<예시 3>`으로 **"1-4-2 (기사-광고 미구분)"**이 명시되어 있습니다. 그러나 바로 위 코드의 `_load_pattern_catalog()` 함수는 API 호출 시 `&is_meta_pattern=eq.false`를 걸어 **메타 패턴(1-4-1, 1-4-2)을 프롬프트 컨텍스트에서 완전히 차단**하고 있습니다.
    - **리스크**: Sonnet은 자신이 제시받은 "패턴 목록"에는 없는 1-4-2 코드를 예시에서 보게 됩니다. 이는 LLM에게 지시 모순(Cognitive Dissonance)을 일으켜, 존재하지 않는 다른 메타 코드나 임의의 코드를 환각(Hallucination)할 위험을 대폭 증가시킵니다.
    - **조치 권고**: 예시 3번을 텍스트 분석이 가능한 다른 정상 소분류 패턴(예: 1-4-3 또는 1-5 계열) 기반으로 교체하십시오.
2. **[PASS] DB 인용 적합성 및 접두어 규칙 준수**
    - DB 마이그레이션 코드를 스캔한 결과, KJA와 같은 혼재된 접두어는 전혀 없이 `JCE`, `JCP`, `PCE` 등으로 규격화되어 있습니다.
    - 또한 `parent_chain` 재귀 CTE (RPC) 안팎으로 `is_citable = TRUE` 필터가 빈틈없이 적용되어 있어 환각 원천 차단 설계는 견고합니다.

### [축 2] 수평 정합성 (모듈 인터페이스 및 시스템)

**판정: ❌ FAIL (조건부)**

데이터클래스 간의 타입 전달이나 파이프라인 흐름(Option B) 자체는 이상이 없습니다. 그러나 **벤치마크 도구의 노후화**와 **네트워크 병목**이 심각합니다.

1. **[CRITICAL] 벤치마크 툴의 왜곡된 지표 산출 (`benchmark_pipeline_v3.py`)**
    - **원인**: 현재 파이프라인은 `match_patterns_solo` (게이트 없는 Sonnet 단독 모델)로 동작함에도 불구하고, 벤치마크 디스플레이 코드는 2-Call 시절의 로직을 그대로 유지하고 있습니다. (예: `suspect_categories`를 읽으려 시도하고 비용 계산 시 Haiku 호출 요금을 인위적으로 합산함)
    - **리스크**: Solo 모드로 전환되면서 Haiku 비용이 전혀 들지 않는데 합산하여 출력하고 있으며, `Haiku 양질 판정`이라는 허위 로그가 찍혀 향후 벤치마크 결과 해석을 왜곡시킵니다.
    - **조치 권고**: v3 스크립트 내부의 불필요한 Haiku 추적/계산 로직을 Sonnet Solo(1-Call) 전용으로 대폭 리팩토링하십시오.
2. **[MAJOR] 벡터 검색 네트워크 IO 병목 (`pattern_matcher.py` - `search_vectors`)**
    - **원인**: `search_pattern_candidates` RPC를 청크 개수만큼 `for`문을 돌며 **동기형(Sequential) httpx.post**로 호출하고 있습니다.
    - **리스크**: 기사가 길어져 청크가 10개가 나오면 API를 10번 순차 대기합니다. M6에서 웹앱에 연동 시 체감 응답시간이 기하급수적으로 느려지는 주범이 됩니다.
    - **조치 권고**: `httpx.AsyncClient`와 `asyncio.gather`를 활용해 벡터 검색 API를 병렬 처리(Concurrent) 하거나, 쿼리 임베딩 배열을 한 번의 RPC 콜로 던지도록 버크(Bulk) 처리 방식으로 개선해야 합니다.
3. **[MAJOR] 파이프라인 전역 에러 핸들링 부재 (`pipeline.py`)**
    - 현재 `CitationResolver` 쪽에만 `try-except`가 걸려 있고, 핵심인 `match_patterns_solo()`나 OpenAI 임베딩 생성 시 발생하는 커넥션 타임아웃/500 에러 등에 대한 방어선이 `analyze_article()`에 존재하지 않습니다. 실패 시 부분적인 리포트 반환 대신 아예 시스템 예외로 터져버리므로, 프로덕션 진입 전 방어 로직 스캐폴딩이 필요합니다.

### [축 3] 코드베이스 위생 (배포 준비 상태)

**판정: ⚠️ MINOR (위생 보강 필요)**

1. **[MINOR] 고아 모듈(Orphan Modules) 잔존**
    - `backend/core/` 하위에 위치한 `analyzer.py`, `criteria_manager.py`, `prompt_builder.py` 모듈은 현재 활성 파이프라인에서 단 1줄도 import를 받지 않는 쓰레기(Dead) 코드입니다. M6 클라우드 배포 시 혼란을 주므로, 런타임에 올라가지 않도록 `_deprecated/` 디렉토리로 격리하거나 삭제하십시오.
2. **[MINOR] `CLAUDE.md` 동기화 지연**
    - 기술 지침서인 `CLAUDE.md`가 여전히 "M4(Haiku→Sonnet 전환 확정)" 시점에 머물러 있습니다. M5의 1-Call 전환 성과와 결정론적 인용 적용 사실, 그리고 v3 벤치마크 스크립트에 대한 현 최신 명령어로 갱신이 시급합니다.

### ■ Antigravity 종합 제언

시스템이 "동작하도록" 만드는 데에는 성공했으나, **모델 혼란 유발(1-4-2 프롬프트)**과 **벤치마크 스크립트 노후화**, **청크 단위 동기화 통신(N+1 Query Issue)**이라는 세 마리 토끼(버그)가 숨죽이고 있었습니다.

Gamnamu님, 위 항목 중 **[CRITICAL] 등급의 2건**은 M6 배포 전 반드시 수정되어야 파이프라인과 벤치마크의 신뢰도가 유지됩니다. 클로드 코드 CLI에게 이 결과를 바탕으로 즉각적인 수정 스크립트(또는 리팩토링)를 요구하시기 바랍니다.
