### M5 성능 목표 — 재정의

M4의 교훈(교훈 15: 저널리즘 비평의 본질적 주관성)을 반영하여, M5는 **절대 수치 목표가 아닌 우선순위 기반 목표**를 설정한다.

| 우선순위 | 지표 | 목표 | 근거 |
|----------|------|------|------|
| **1순위** | TN FP Rate | ≤ 33% (6건 중 2건 이하) | 양질의 보도 오판 방지가 최우선. M4: 67%(4/6) |
| **2순위** | Category Recall | ≥ 60% | 대분류 수준에서 문제 영역을 포착. M4: 46.7% |
| **3순위** | Final Recall | ≥ 40% | 소분류 정확도는 점진 개선. M4: 28.3% |
| **참고** | Final Precision | ≥ 30% | Recall 개선 시 동반 하락 가능, 모니터링 |
| **참고** | Candidate Recall | 50.8%+ | 벡터 검색 구조 변경은 M5 후순위 |

### M5 핵심 전략: Few-shot이 캘리브레이션의 열쇠

M4 교훈 12에서 입증: E-12가 few-shot 예시(B-01)와 유사한 인권·차별 기사에서 FR 1.00을 달성.
현재 2건(TP 1 + TN 1) → 대분류별 대표 케이스로 확대하면 LLM의 판단 기준이 정교화된다.

### STEP 51. Claude.ai — Few-shot 케이스 선정 전략 확정

**[배경]** 현재 few-shot은 2건(B-01 TP, C2-07 TN)뿐이다.
M4 벤치마크에서 대분류별 성능 편차가 크므로(1-4: 25%, 1-8: 33% vs 1-3: 42%), 대분류별 대표 케이스가 필요.
단, few-shot 예시가 벤치마크 대상이면 성능 과대평가 위험이 있다(M4 주의사항).

**[설계안]**

Few-shot 선정 원칙:
1. **대분류별 1건씩**: 1-1, 1-3, 1-4, 1-5, 1-7, 1-8 각 1건 = 6건 TP
2. **TN 추가 2건**: 현재 1건(C2-07) → 3건으로 확대. 특히 인권·정치 주제의 양질 보도
3. **총 9건** (TP 6 + TN 3)
4. **벤치마크 과대평가 방지**: few-shot 예시 9건은 벤치마크에서 **제외**하고, 나머지 17건으로 벤치마크 실행
5. **예시 형식**: 기사 제목 + 요약(2~3문장) + 올바른 분석 JSON (기사 전문은 토큰 예산상 불포함)

### STEP 52. Claude.ai — TN 구분 강화 전략 확정

**[설계안]**

1. **TN Few-shot 확대** (STEP 51에서 다룸): C-02, E-19를 TN 예시로 추가
2. **1단계 품질평가 프롬프트 강화**: 
   - "인권·사회 문제를 다루는 탐사보도·기획보도는 강렬한 표현이 취재 결과에 부합하면 양질의 보도"

   - 프롬프트에 "인권, 차별, 성소수자, 정치 관련 기사는 특히 신중하게 1단계를 수행하세요" 추가

### STEP 55. Claude.ai — Few-shot 예시 9건 초안 작성

**[작업]** STEP 51에서 확정된 9건에 대해, 프롬프트에 삽입할 few-shot 예시를 작성한다.

**[예시 형식 — 건당]**
```
### 예시 N — [TP/TN] [대분류명]
기사 제목: "..."
기사 요약: (2~3문장, 핵심 내용과 문제점/양질 포인트)
올바른 분석:
```json
[
  {
    "matched_text": "구체적 문장 인용",
    "reasoning": "판단 근거 2~3문장",
    "severity": "high/medium/low",
    "pattern_code": "1-X-Y"
  }
]
```
```

**[작성 원칙]**
1. 기사 전문이 아닌 **요약**(2~3문장)으로 토큰 절약
2. `matched_text`는 실제 기사에서 발견되는 표현을 정확히 인용
3. `reasoning`은 **왜 이 패턴인지, 왜 다른 유사 패턴이 아닌지** 명시
4. TN 예시는 "왜 이 기사가 양질인지"를 reasoning에 상세히 기술
5. 혼동 쌍(STEP 51 참고) 교정에 특히 유의

**[산출물]** `docs/M5_FEW_SHOT_EXAMPLES.md` — 9건의 예시 원문

**[감리 판정 기준]**
- ✅ 9건 모두 작성되었는가
- ✅ 예시별 reasoning이 혼동 패턴 구분에 도움이 되는가
- ✅ TN 예시 3건이 "양질의 보도 기준"을 명확히 제시하는가
- ✅ 토큰 추정: 예시 9건 총 ~2,000~3,000 토큰 이내인가

## 비상 시나리오 (Emergency Scenarios)
## ━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

### 비상 M: Few-shot 확대 후 토큰 예산 초과

**[상황]** 9건 few-shot + 28개 패턴 목록 + 기사 전문 합산이 Sonnet 입력 한도에 근접
**[대응]**
1. few-shot 예시의 `reasoning`을 1~2문장으로 압축
2. 패턴 목록에서 description을 제거하고 code+name만 유지
3. 그래도 부족하면 few-shot을 6건(TP 4 + TN 2)으로 축소
4. 최악의 경우: 장문 기사(5000자+)에서만 few-shot을 축약 버전으로 전환

### 비상 N: TN 강화가 TP Recall을 심각하게 하락시킴

**[상황]** TN FP Rate는 개선되었으나 Final Recall이 M4 대비 10%p 이상 하락
**[대응]**
1. 1단계 품질평가 문구의 강도를 단계적으로 완화
2. "인권·정치 기사만" 신중 판단 → "전체 기사에 대해" 신중 판단으로 범위가 확대되었는지 확인
3. TN 강화 문구를 1단계 끝이 아닌 few-shot TN 예시 내부에만 배치하는 방식 검토
4. 극단적 경우: TN 3건 vs TP 20건의 가중치를 벤치마크에서 조정

### 비상 O: CitationResolver에서 DB 조회 대량 실패

**[상황]** Sonnet이 출력하는 ref 코드가 DB ethics_codes의 code와 체계적으로 불일치
**[대응]**
1. Sonnet 프롬프트의 cite 지시를 확인 — ref에 넣어야 할 값이 code인지 id인지
2. 코드 형식 불일치 원인 파악: "JCE-03" vs "JCE-03-01" vs "기자윤리강령 제3조" 등
3. 퍼지 매칭 도입 검토 (LIKE 쿼리 또는 prefix 매칭)
4. Sonnet 프롬프트에 "ref는 반드시 ethics_codes 테이블의 code 컬럼 값과 동일해야 함" 명시

### 비상 P: 벤치마크 17건이 통계적으로 불충분

**[상황]** 9건 제외 후 17건으로는 성능 판단의 신뢰도가 낮다는 감리 의견
**[대응]**
1. 26건 전체 결과를 보조 지표로 활용 (few-shot 과대평가 감안)
2. Reserved Test Set 73건 중 10건을 **1회성 검증용**으로 추출 (개발에 재사용 금지)
3. 교차 검증: few-shot 예시를 2그룹(A/B)으로 나누어 각각 벤치마크 → 편차 확인
4. 최종적으로 M5 성능은 "17건 공식 + 26건 참고 + 10건 홀드아웃" 3중 보고
