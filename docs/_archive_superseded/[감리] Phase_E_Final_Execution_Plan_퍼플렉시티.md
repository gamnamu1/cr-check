# 퍼플렉시티

## 감리 의견 및 개선 제안

**1. `requirements.txt` 누락 패키지 처리 (E-1-1) - 완벽한 조치**

- `httpx`와 `openai`가 파이프라인의 핵심 모듈에서 사용됨에도 누락된 것을 찾아낸 것은 치명적인 서버 다운을 막은 훌륭한 조치입니다.
- Railway 환경에서 빌드 에러를 유발하기 쉬운 C 라이브러리 의존성(`weasyprint`)을 과감히 제거한 판단도 배포 성공률을 크게 높일 것입니다.

**2. `db.py` 클라우드 전환 로직 지연 해결 (E-1-2) - 매우 중요**

- 매 API 호출마다 5초의 지연이 발생하는 병목을 찾아내어 조건 분기로 최적화한 것은 실서비스 사용성에 직결되는 아주 중요한 개선입니다.
- 제안된 수정안 코드가 논리적으로 완벽하므로 그대로 적용하시면 됩니다.

**3. 배포 순서의 엄격성 (E-2 → E-4) - 핵심 포인트**

- **코드 배포 전 DB 마이그레이션 선행** 원칙을 명시한 점이 가장 중요합니다. `share_id` (NOT NULL) 컬럼이 없는 상태에서 새 코드가 배포되면 INSERT 에러가 발생하므로, 계획서에 명시된 순서(E-2 → E-4)를 반드시 엄수해야 합니다.

**4. 추가 제안 사항 (선택적 고려)**

- **CORS 프리뷰 URL:** 현재 Vercel의 브랜치 프리뷰 URL이 제외되어 있습니다. 당장 `main` 배포에는 문제가 없으나, 추후 `feature/m6-wip` 단계에서 프론트엔드 프리뷰를 테스트하려면 `main.py`의 `allow_origins`에 정규식이나 와일드카드를 활용해 Vercel 프리뷰 도메인(`.vercel.app`)을 포괄적으로 열어주는 것을 고려해 볼 수 있습니다.
- **DB 마이그레이션 백업:** `analysis_results` 테이블 마이그레이션(Phase D)을 SQL Editor에서 실행하기 전, 만약을 대비해 기존 `analysis_results` 테이블의 스키마 DDL만이라도 로컬에 메모해 두는 것이 안전합니다 (데이터가 0건이므로 큰 위험은 없습니다).

계획서가 이미 매우 구체적이고 안전 장치들이 잘 마련되어 있어, 이대로 진행하시면 성공적인 Phase E 클라우드 배포가 될 것으로 보입니다.
