# DB 구축 직후 품질 저하 - 원인 진단 및 개선안_Gemini

상위 항목: DB 구축 직후 품질 저하 - 원인 진단 및 개선안 (https://www.notion.so/DB-33766d0370c680339dc4dce5b3625194?pvs=21)

보내주신 파이프라인 가이드와 실제 백엔드 파이썬 코드들을 꼼꼼히 교차 검증했습니다. `pipeline.py`를 중심으로 ❰0❱ 스크래핑부터 ❰5❱ 결정론적 인용 후처리까지 이어지는 흐름을 완벽히 파악했습니다.

코드를 살펴보니, 만족도가 80%에서 50% 이하로 급락한 원인이 단순한 프롬프트 톤의 문제가 아니라, **파이프라인 단계별 미세한 설정 값과 후처리 로직의 부작용**일 가능성이 강력하게 의심됩니다.

특히 코드상에서 당장 눈에 띄는 **잠재적 누수 지점(Leak Points) 3가지**를 발견했습니다. 진단 아이디어와 함께 공유해 드립니다.

### 🚨 코드에서 발견된 강력한 의심 지점 3가지

1. **벡터 검색의 병목 (설계와 구현의 불일치)**
    - **의심점**: `마스터 플랜(v4.0)`에서는 무관한 패턴이 섞이더라도 정답을 놓치지 않기 위해 후보를 10개(Top-K=10), Threshold 0.5로 설정하기로 했습니다. 하지만 실제 `pattern_matcher.py`를 보면 `VECTOR_MATCH_COUNT = 7`, `VECTOR_THRESHOLD = 0.2`로 매우 타이트하게 조여져 있습니다.
    - **문제**: 여기서 정답 패턴이 7위 안에 들지 못해 잘려나가면, 뒤에 있는 Sonnet은 아무리 똑똑해도 정답을 맞출 수 없습니다.
2. **Citation Resolver의 규범 원문 절단 (Truncation)**
    - **의심점**: `citation_resolver.py` 코드를 보면 `_truncate_text(text: str, max_len: int = 200)` 함수가 존재합니다.
    - **문제**: 기존 평가에서는 윤리규범이 온전히 인용되었지만, 현재는 `<cite>` 태그 치환 시 원문이 200자 내외에서 강제로 잘릴 가능성이 있습니다. 이로 인해 리포트에서 규범의 핵심 맥락이 소실되어 "윤리규범을 정확히 인용하지 못한다"는 불만이 나올 수 있습니다.
3. **Sonnet Solo의 과도한 자기 검열 (Devil's Advocate CoT)**
    - **의심점**: 1차 판단을 내릴 때 AI가 스스로 반론을 제기하며 깐깐하게 평가하도록 CoT가 적용되어 있습니다.
    - **문제**: 이 로직이 너무 엄격하게 작동하여, 예전 같으면 잡아냈을 명백한 위반 사항을 '허용되는 관행'이라며 스스로 기각(과소 탐지)하고 있을 수 있습니다.

### 🔬 진단 및 해법 도출을 위한 'X-Ray' 테스트 제안

어디서 문제가 발생했는지 명확히 가려내기 위해, **기사 1~2건을 선정하여 파이프라인의 각 관문을 통과할 때마다 데이터를 열어보는 'X-Ray 진단'**을 제안합니다.

만족스럽지 못했던 기사 URL 하나를 로컬에서 실행하신 후, 터미널 로그나 디버그 출력을 통해 다음 **4가지 데이터**를 제게 던져주시면 됩니다.

1. **[관문 1: Vector Search 결과]**
    - `pattern_matcher.py`의 `candidates` 값 (검색된 7개 패턴 리스트)
    - *진단 포인트*: 정답 후보가 7위 안에 제대로 올라왔는가?
2. **[관문 2: 1차 식별 결과]**
    - `pattern_matcher.py`의 `detections` (Sonnet Solo가 반환한 JSON)
    - *진단 포인트*: 후보에 있었는데도 Sonnet이 기각해버렸는가? 아니면 엉뚱한 걸 골랐는가?
3. **[관문 3: 매핑된 규범]**
    - `report_generator.py`로 넘어가는 `ethics_refs` 데이터
    - *진단 포인트*: 식별된 패턴에 맞는 적절한 규범 원문이 DB에서 잘 딸려 왔는가?
4. **[관문 4: 치환 전후의 리포트 본문]**
    - `<cite ref="..."/>`가 들어있는 Sonnet의 원본 출력과, `citation_resolver`를 거친 최종 출력 비교
    - *진단 포인트*: 리포트의 말투(톤)가 이상한가? 아니면 Resolver가 텍스트를 망가뜨렸는가?

이 프로세스에 동의하신다면, **평가 결과가 좋지 않았던 기사(URL 또는 원문)와 해당 기사를 로컬에서 돌렸을 때의 백엔드 로그(관문 1~4)**를 공유해 주시겠어요? 제가 즉시 누수 지점을 핀포인트로 찾아내고 프롬프트나 파라미터 수정안을 제시하겠습니다.
