### 2-2. Phase 1 기본 테이블 동시 생성 (Auth 제거 반영 확정 SQL)

아래 SQL을 그대로 실행한다. 아카이브 문서(`DB_CONSTRUCTION_FINAL_PLAN.md`)에는
Auth 시절 코드가 남아있으므로 **참조하지 말 것**. 이 섹션의 SQL이 확정본이다.

```sql
-- ============================================
-- CR-Check Phase 1 테이블 (Auth 제거 반영)
-- 전 Phase 익명 운영, 공개 URL 공유
-- ============================================

-- 1. Articles (기사 원본 메타데이터)
-- 목적: 중복 분석 방지, 언론사별 통계 기반
CREATE TABLE public.articles (
  id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  url TEXT NOT NULL,
  title TEXT NOT NULL,
  publisher TEXT,
  journalist TEXT,
  publish_date TIMESTAMPTZ,
  created_at TIMESTAMPTZ DEFAULT now() NOT NULL,

  CONSTRAINT articles_url_key UNIQUE (url)
);

CREATE INDEX idx_articles_url ON public.articles(url);
CREATE INDEX idx_articles_publisher ON public.articles(publisher);
CREATE INDEX idx_articles_created_at ON public.articles(created_at DESC);

COMMENT ON TABLE public.articles IS '기사 원본 메타데이터 (중복 분석 방지용)';


-- 2. Analysis Results (분석 결과)
-- 목적: AI 분석 결과 저장 + 공개 URL(/report/{id})로 접근·공유
-- Auth 제거: user_id 없음, 모든 리포트는 익명·공개
CREATE TABLE public.analysis_results (
  id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  article_id BIGINT REFERENCES public.articles(id) ON DELETE CASCADE NOT NULL,

  -- 리포트 내용
  comprehensive_report TEXT,
  journalist_report TEXT,
  student_report TEXT,

  -- 메타데이터
  phase1_model TEXT DEFAULT 'claude-haiku-4-5-20251001',
  phase2_model TEXT DEFAULT 'claude-sonnet-4-6',
  duration_seconds FLOAT,
  detected_categories JSONB,   -- Phase 1 결과 (예: ["1-1-1", "1-7-3"])

  created_at TIMESTAMPTZ DEFAULT now() NOT NULL
);

CREATE INDEX idx_analysis_article ON public.analysis_results(article_id);
CREATE INDEX idx_analysis_created_at ON public.analysis_results(created_at DESC);

COMMENT ON TABLE public.analysis_results IS 'AI 분석 결과 저장 (익명, 공개 URL 공유)';
COMMENT ON COLUMN public.analysis_results.detected_categories IS 'Phase 1에서 탐지된 문제 카테고리 ID 목록';


-- 3. Feedbacks (익명 사용자 피드백) — Phase 3에서 활성화
-- Auth 제거: user_id 없음, analysis_id + rating + comment만 저장
CREATE TABLE public.feedbacks (
  id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  analysis_id BIGINT REFERENCES public.analysis_results(id) ON DELETE CASCADE NOT NULL,
  rating INT CHECK (rating BETWEEN 1 AND 5),
  comment TEXT,
  created_at TIMESTAMPTZ DEFAULT now() NOT NULL
);

CREATE INDEX idx_feedbacks_analysis ON public.feedbacks(analysis_id);

COMMENT ON TABLE public.feedbacks IS '익명 사용자 피드백 (user_id 없이 analysis_id + rating + comment만 저장)';
```

> **RLS 참고**: 전 Phase 익명 운영이므로 RLS를 비활성화 상태로 유지한다.
> 백엔드(FastAPI)는 `SERVICE_ROLE_KEY`로 접근하여 RLS를 우회하므로,
> 별도 정책 설정 없이 운영 가능하다. 향후 사용자 인증 도입 시 RLS 정책을 추가한다.

이후 v4.0 섹션 6.1의 `analysis_ethics_snapshot` 테이블도 생성 가능 (analysis_results에 의존).

**Phase 3 통계 함수** (Phase 2 대시보드 구현 시 실행):

```sql
-- 실시간 트렌드 (최근 N시간 내 인기 기사)
CREATE OR REPLACE FUNCTION get_trending_articles(hours_ago INT DEFAULT 24)
RETURNS TABLE (
  article_id BIGINT,
  title TEXT,
  url TEXT,
  publisher TEXT,
  analysis_count BIGINT,
  latest_analysis TIMESTAMPTZ
) AS $$
BEGIN
  RETURN QUERY
  SELECT a.id, a.title, a.url, a.publisher,
         COUNT(ar.id) as analysis_count,
         MAX(ar.created_at) as latest_analysis
  FROM public.articles a
  JOIN public.analysis_results ar ON a.id = ar.article_id
  WHERE ar.created_at >= NOW() - INTERVAL '1 hour' * hours_ago
  GROUP BY a.id, a.title, a.url, a.publisher
  ORDER BY analysis_count DESC, latest_analysis DESC
  LIMIT 10;
END;
$$ LANGUAGE plpgsql STABLE;

-- 언론사별 통계
CREATE OR REPLACE FUNCTION get_publisher_stats()
RETURNS TABLE (
  publisher TEXT,
  total_analyses BIGINT,
  avg_duration FLOAT
) AS $$
BEGIN
  RETURN QUERY
  SELECT a.publisher, COUNT(ar.id), AVG(ar.duration_seconds)
  FROM public.articles a
  JOIN public.analysis_results ar ON a.id = ar.article_id
  WHERE a.publisher IS NOT NULL
  GROUP BY a.publisher
  ORDER BY total_analyses DESC
  LIMIT 20;
END;
$$ LANGUAGE plpgsql STABLE;

-- 전체 통계 요약 (Auth 제거: unique_users 항목 제거)
CREATE OR REPLACE FUNCTION get_overall_stats()
RETURNS TABLE (
  total_articles BIGINT,
  total_analyses BIGINT,
  avg_duration FLOAT
) AS $$
BEGIN
  RETURN QUERY
  SELECT COUNT(DISTINCT a.id),
         COUNT(ar.id),
         AVG(ar.duration_seconds)
  FROM public.articles a
  LEFT JOIN public.analysis_results ar ON a.id = ar.article_id;
END;
$$ LANGUAGE plpgsql STABLE;
```

```python
# seed_ethics.py (개념적 예시)
# ⚠️ 2-패스 삽입: JSON의 parent_code_id는 문자열("JEC-8")이지만
#    DB 컬럼은 BIGINT이므로, 1차에서 NULL로 삽입 후 2차에서 매핑.
#    (앙상블 검증 2026-03-25 반영)
mapping = json.load(open("ethics_codes_mapping.json"))

# 패스 1: parent_code_id를 NULL로 전체 INSERT → ID 확보
for ec in mapping:
    supabase.table("ethics_codes").insert({
        "code": ec["code"],
        "title": ec["title"],
        "full_text": ec["full_text"],
        "source": ec["source"],
        "article_number": ec["article_number"],
        "tier": ec["tier"],
        "tier_rationale": ec.get("tier_rationale"),
        "parent_code_id": None,  # 1차에서는 NULL
        "is_citable": ec.get("is_citable", True),  # 서문/부칙은 False
        "domain": ec["domain"],
        "locale": "ko-KR",
        "version": 1,
        "is_active": True,
        "effective_from": "2026-03-15"
    }).execute()

# 패스 2: code→id 매핑 딕셔너리 생성 후 parent UPDATE
rows = supabase.table("ethics_codes").select("id, code").execute()
code_to_id = {r["code"]: r["id"] for r in rows.data}

for ec in mapping:
    if ec.get("parent_code_id"):
        parent_id = code_to_id[ec["parent_code_id"]]
        supabase.table("ethics_codes").update({
            "parent_code_id": parent_id
        }).eq("code", ec["code"]).execute()
```

### 4-2. 임베딩 생성 스크립트

```python
# generate_embeddings.py (개념적 예시)
import openai
from supabase import create_client

openai_client = openai.OpenAI(api_key=OPENAI_API_KEY)
supabase = create_client(SUPABASE_URL, SUPABASE_SERVICE_ROLE_KEY)

# 패턴 임베딩 (배치 처리)
patterns = supabase.table("patterns").select("id, description").execute()
texts = [p["description"] for p in patterns.data]

# 배치 API 1회 호출로 모든 패턴 임베딩 생성
response = openai_client.embeddings.create(
    input=texts,
    model="text-embedding-3-small"
)

for i, p in enumerate(patterns.data):
    supabase.table("patterns").update({
        "description_embedding": response.data[i].embedding
    }).eq("id", p["id"]).execute()

# 규범 임베딩 (동일 방식)
ethics = supabase.table("ethics_codes").select("id, full_text").eq("is_active", True).execute()
# ... (동일 패턴)
```

> 262개 임베딩 생성 비용: ~$0.01 미만
> 소요 시간: ~10초 (배치 1회)

## 5단계: 골든 데이터셋 구축 + 벤치마크

### 5-1. 골든 데이터셋 20~30건 수동 선정

8개 대분류에서 각 3~4건의 실제 기사를 선정.
각 기사에 대해 "이 기사에서 반드시 탐지되어야 할 패턴"을 사람이 판단.

```json
// golden_dataset.json 구조 예시
[
  {
    "id": "GD-001",
    "article_url": "https://example.com/news/12345",
    "article_key_text": "정부 고위 관계자에 따르면...(기사 핵심 문단)",
    "expected_patterns": ["1-1-1", "1-2-1"],
    "expected_ethics_codes": ["KJA-3", "PCC-7.2"],
    "difficulty": "easy",
    "category": "진실성",
    "notes": "익명 단일 취재원, 반론 없음"
  },
  ...
]
```

### 5-2. 임베딩 모델 벤치마크 (선택적)

골든셋의 article_key_text를 쿼리로,
`search_pattern_candidates()` 함수의 결과에 expected_patterns가 포함되는지 측정.

```python
# benchmark.py (개념적 예시)
for case in golden_dataset:
    embedding = generate_embedding(case["article_key_text"])
    results = supabase.rpc("search_pattern_candidates", {
        "query_embedding": embedding,
        "match_threshold": 0.5,
        "match_count": 10
    }).execute()

    found_codes = {r["pattern_code"] for r in results.data}
    expected = set(case["expected_patterns"])
    recall = len(found_codes & expected) / len(expected)
    # ... 집계
```

### 5-3. threshold 튜닝

벤치마크 결과를 보고 threshold를 조정:
- Recall이 낮으면 → threshold를 낮춤 (0.5 → 0.4)
- 노이즈가 많으면 → threshold를 높이거나 match_count를 줄임

## 백엔드 구현 시 참조사항 (2026-03-22 추가)

> 4개 AI 리뷰어(Gemini, NotebookLM, Perplexity, Manus)의 `ethics_codes_mapping.json` 교차 검수에서 도출된 사항.
> JSON 파일 자체는 수정 불필요. **M2(스키마 생성)~M3(시드 데이터 입력) 단계에서 Claude Code CLI가 반영해야 할 항목.**

### 참조 1: 초단문 엔트리의 임베딩 결합 전략

`full_text`가 30자 미만인 엔트리가 26개 있음 (SRE 계열에 집중). 짧은 명사구는 벡터 검색에서 문맥 부족으로 누락(false negative)될 수 있음.

- **원칙**: `full_text`는 Deterministic Quoting을 위해 규범 원문 그대로 보존. 절대 윤문하지 않음.
- **구현**: 임베딩 벡터 생성 시 `title + " — " + full_text`를 결합한 파생 텍스트를 사용.
- **적용 대상**: M4(임베딩 생성) 단계의 시드 스크립트 (`migrate_criteria.py` 또는 해당 모듈).

### 참조 2: 서문/운영 조항의 검색 제외 플래그

서문(16개, code가 `-P`/`-T1`/`-T2`로 끝남)과 부칙 성격 조항(DRG-40~43, EPG-27~29 등)은 기사 분석 시 위반 근거로 인용될 가능성이 낮아 벡터 검색에서 노이즈로 작용할 수 있음.

- **구현**: `ethics_codes` 테이블에 `is_citable BOOLEAN DEFAULT TRUE` 컬럼 추가. 서문/부칙 엔트리는 `FALSE`로 설정.
- **활용**: 검색 쿼리의 WHERE 절에서 `is_citable = TRUE` 필터 적용. 단, 롤업 리포트에서 맥락 설명용으로는 참조 가능해야 하므로 데이터 자체는 삭제하지 않음.
- **적용 대상**: M2(스키마 생성) + M3(시드 데이터 입력).

### 참조 3: Junction 배열의 DB 분리 삽입

JSON의 `junction` 배열(42개 엔트리 보유, 각각 `parent_code_id` + `context_hint`)은 `ethics_code_hierarchy` 테이블로 분리 INSERT해야 함. 이는 `DB_AND_RAG_MASTER_PLAN_v4.0`에 이미 명시된 사항이나 재확인 차원에서 기록.

- **구현**: 시드 스크립트에서 각 엔트리의 `junction` 배열을 파싱하여, primary parent는 `ethics_codes.parent_code_id` 컬럼에, 보조 부모는 `ethics_code_hierarchy` 테이블에 `(code, parent_code_id, context_hint)` 형태로 INSERT.
- **적용 대상**: M3(시드 데이터 입력).

## 주의사항

1. **무료 티어 일시중지**: 1주 미사용 시 자동 pause. 개발 기간 중 주기적으로 접속하거나,
   간단한 cron job(예: 매일 1회 health check API 호출)으로 방지.
2. **SERVICE_ROLE_KEY 보안**: 이 키는 RLS(Row Level Security)를 우회하므로
   절대 프론트엔드에 노출 금지. 백엔드 `.env`에만 저장.
3. **RLS 정책**: 전 Phase 익명 운영이므로 RLS를 비활성화하거나
   permissive 정책으로 설정. 향후 사용자 인증 도입 시 적절한 정책으로 전환.
4. **백업**: 무료 티어는 자동 백업이 없음. 중요한 시드 데이터 입력 전후로
   SQL Export(Dashboard > Database > Backups)를 수동으로 수행.
