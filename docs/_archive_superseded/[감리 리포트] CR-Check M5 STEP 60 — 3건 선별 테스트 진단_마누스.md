# [감리 리포트] CR-Check M5 STEP 60 — 3건 선별 테스트 진단_마누스

네, Max 모델로서 STEP 61 감리 요청서와 첨부된 모든 관련 문서를 심층적으로 분석했습니다. 3건의 선별 테스트가 모두 실패로 돌아간 것은 실망스러운 결과이지만, 동시에 현재 아키텍처의 근본적인 한계를 명확하게 드러내는 매우 중요한 데이터입니다.

이것은 '감리 협의'를 넘어, **아키텍처의 근본적인 재설계(Radical Redesign)**가 필요하다는 강력한 신호입니다.

---

### **1. 독립적 원인 진단: '이중 임피던스 불일치'의 발생**

M4 감리에서 제가 '의미론적 임피던스 불일치'를 지적했습니다. 이번 3건의 테스트 실패는 그 문제가 해결되지 않았을 뿐만 아니라, **새로운 불일치가 추가되어 '이중 임피던스 불일치' 상태**에 빠졌음을 보여줍니다.

1. **1차 불일치 (벡터 검색 ↔ LLM):**
    - **현상:** `E-15` 케이스에서 `Candidate Recall`이 0%를 기록했습니다. 이는 기사 텍스트의 '의미'와 패턴 `description`의 '의미'가 벡터 공간에서 너무 멀리 떨어져 있어, 벡터 검색이 애초에 정답 후보조차 찾아오지 못하는 문제입니다.
    - **본질:** 이것이 바로 **"의미론적 임피던스 불일치"**입니다. 서로 다른 종류의 언어(구체적 사건 vs 추상적 규범)를 같은 잣대로 비교하려는 구조적 한계입니다. `VECTOR_MATCH_COUNT`를 조정하는 것은 이 문제를 해결하지 못합니다.
2. **2차 불일치 (품질 평가 ↔ 패턴 식별):**
    - **현상:** `B2-10` 케이스에서, 1단계 품질평가 모듈이 "이 기사는 양질의 보도"라고 **성급하게 오판**하여, 2단계 패턴 식별 단계로 넘어가지도 못하고 분석이 종료되었습니다.
    - **본질:** 이는 **"작업의 임피던스 불일치"**입니다. 현재 프롬프트는 LLM에게 두 가지 매우 다른 종류의 작업을 동시에 요구합니다.
        - **1단계 (숲 보기):** 기사 전체의 논조, 공익성, 균형감 등 **거시적이고 주관적인 품질**을 평가.
        - **2단계 (나무 보기):** 기사 내 특정 문장, 특정 표현의 **미시적이고 구체적인 위반**을 식별.
    - `B2-10`은 숲(전체적인 팩트체크 형식)은 그럴듯해 보이지만, 나무(세부적인 통계 비교 방식)가 썩어있는 경우입니다. 현재의 2단계 프롬프트는 LLM이 숲만 보고 "문제없다"고 판단하게 만들어, 나무를 자세히 들여다볼 기회 자체를 박탈합니다.

**결론:** 현재 시스템은 **입력단(벡터 검색)과 처리단(LLM 프롬프트) 양쪽에서 모두 구조적인 한계**에 부딪혔습니다. Few-shot 예시를 더 추가하거나 문구를 조정하는 '땜질식 처방'으로는 이 문제를 해결할 수 없습니다.

### **2. 개선 아이디어: '2단계 파이프라인'의 전면 도입**

M4 감리 때 제가 제안했던 **'2단계 파이프라인'** 아이디어를 이제 M5에서 **전면적으로 도입해야 할 때**입니다. 현재의 '한 프롬프트 내 2단계 구조'가 실패했음이 명백해진 이상, 이를 **물리적으로 분리된 2개의 LLM 호출(2-Call Pipeline)**로 재설계해야 합니다.

### **Max 제안: '의심-검증(Suspect-Verify)' 파이프라인**

이 새로운 파이프라인은 저널리즘 비평의 실제 과정, 즉 '어딘가 이상한데?(의심)' → '구체적으로 뭐가 문제지?(검증)'의 흐름을 모방합니다.

**1단계: 의심 단계 (Suspect Stage) - 대분류 식별기**

- **모델:** Claude **Haiku** (비용 효율적)
- **입력:** 기사 전문
- **프롬프트:**
    - "이 기사는 아래 8개 대분류 중 어떤 영역에서 **문제가 될 소지가 가장 높은가?** 1~3개만 골라라. 확신이 없으면 `['None']`을 반환하라."
    - (Few-shot 예시는 여기에 집중적으로 사용하여, 대분류를 판단하는 기준을 학습시킵니다.)
- **역할:** '품질 평가'라는 모호한 작업 대신, **'가장 의심스러운 문제 영역 식별'**이라는 더 구체적이고 쉬운 작업을 수행합니다. 이진 게이트가 아니라, 분석의 '초점'을 정해주는 역할을 합니다.
- **출력 예시:** `['1-1', '1-7']`

**2단계: 검증 단계 (Verify Stage) - 소분류 확정기**

- **모델:** Claude **Sonnet** (정교한 판단)
- **입력:** 기사 전문 + **1단계에서 선택된 대분류에 속하는 소분류 목록만** (예: 1-1 계열 5개, 1-7 계열 6개) + 벡터 검색 후보(★)
- **프롬프트:**
    - "1단계 분석 결과, 이 기사는 '진실성(1-1)'과 '언어(1-7)' 영역에서 문제가 의심된다. 아래의 관련 소분류 목록과 기사를 비교하여, **실제로 위반이 확인되는 패턴만** 정확히 찾아내라."
- **역할:** 제한된 선택지(5~10개) 내에서만 집중적으로 분석하므로, '소분류 혼동' 문제가 획기적으로 개선되고 '패턴 식별'이라는 본연의 임무에만 집중할 수 있습니다.
- **출력 예시:** 최종 패턴 분석 JSON

**이 구조가 왜 지금 문제의 해결책인가?**

1. **'이중 임피던스 불일치' 해소:**
    - **작업 불일치 해소:** '숲 보기(Haiku)'와 '나무 보기(Sonnet)' 작업을 물리적으로 분리하여, 각 모델이 자신의 장점에 맞는 명확한 단일 과업만 수행하게 합니다.
    - **의미 불일치 완화:** 벡터 검색의 역할이 '정답 후보 제시'에서 '2단계 검증 과정의 참고 자료'로 격하되므로, `Candidate Recall` 실패의 영향력이 줄어듭니다.
2. **`B2-10`과 `E-19` 문제 동시 해결:**
    - `B2-10`: 1단계 Haiku가 '팩트체크' 형식을 보고 `['1-1']`을 의심 영역으로 식별하면, 2단계 Sonnet은 1-1 계열 패턴들을 집중 검토하여 `1-1-5`를 찾아낼 수 있습니다. 더 이상 '양질'이라는 성급한 판단으로 분석이 중단되지 않습니다.
    - `E-19`: 1단계 Haiku가 '탐사보도'의 특성을 학습하여 `['None']`을 반환하면, 2단계 자체가 실행되지 않아 FP(오탐)가 원천적으로 차단됩니다.

### **3. "저널리즘 비평의 주관성"과 성능 한계**

현재의 성능(0건 개선)은 '주관성'으로 수용할 수 있는 범위를 명백히 벗어났습니다. 이는 시스템이 아직 '비평'의 영역에 도달하지 못하고, '오작동'의 영역에 머물러 있음을 의미합니다.

'의심-검증' 파이프라인을 도입한 후에도 여전히 해결되지 않는 문제들이 발생한다면, 그때 가서야 비로소 우리는 '저널리즘 비평의 본질적 주관성'과 'AI의 기술적 한계'에 대해 논할 자격이 생길 것입니다. 지금은 아키텍처의 구조적 결함을 먼저 해결하는 데 집중해야 합니다.

**결론: 현재의 2단계 프롬프트 구조를 폐기하고, '의심-검증(Suspect-Verify)' 2-Call 파이프라인으로 아키텍처를 즉시 재설계할 것을 강력히 권고합니다.** 이것이 M5를 성공으로 이끌 유일한 길입니다.
