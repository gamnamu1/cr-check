M1부터 M5까지의 전체 코드베이스에 대한 종합 감리를 완료했습니다. 이 감리는 개별 모듈의 기능 검증을 넘어, 파이프라인 전체의 **수직/수평 정합성**과 **배포 후 유지보수성**이라는 구조적 관점에서 진행되었습니다.

**종합 판정: 조건부 통과 (Conditional Pass)**

프로젝트는 M5 목표를 달성하며 핵심 기능 구현에 성공했습니다. 그러나, 배포 전에 반드시 해결해야 할 **2건의 CRITICAL 이슈**와, 장기적 안정성을 위해 강력히 권고하는 **3건의 MAJOR 이슈**를 식별했습니다. 이 문제들을 해결하지 않고 배포를 강행할 경우, 예측 불가능한 런타임 오류와 심각한 유지보수 부채에 직면하게 될 것입니다.

### **[축 1] 수직 정합성 (DB ↔ 코드 ↔ 프롬프트)**

**판정: FAIL**

### **CRITICAL: `get_ethics_for_patterns` RPC의 암묵적 의존성 및 데이터 불일치**

- **문제점:** `report_generator.py`는 `get_ethics_for_patterns` DB 함수(RPC)를 호출하여 패턴에 연결된 규범을 가져옵니다. 그러나 이 RPC는 `pattern_ethics_relations` 테이블의 `is_active = true`인 관계만 반환하도록 설계되어 있습니다. (`create_cr_check_schema.sql` 273행)
- **위험성:** 현재 `seed_data.sql`에는 `is_active`가 `false`인 관계가 10건 포함되어 있습니다. 만약 Sonnet이 이 10건 중 하나에 해당하는 패턴을 감지하면, `get_ethics_for_patterns`는 **아무런 규범도 반환하지 않을 것**입니다. 이 경우 `report_generator.py`는 빈 `ethics_context`를 가지고 Sonnet을 호출하게 되어, 리포트 품질이 심각하게 저하되거나 Sonnet이 환각을 일으킬 수 있습니다. 이는 **데이터 레벨에서 발생하는 조용한 실패(Silent Failure)**로, 디버깅이 매우 어렵습니다.
- **1차 감리가 놓친 이유:** 이 문제는 SQL, 시드 데이터, Python 코드를 모두 넘나들며 확인해야 하는 복합적인 문제입니다. 개별 모듈만 봐서는 발견하기 어렵습니다.
- **개선 제안 (필수):**
    1. `report_generator.py`의 `fetch_ethics_for_patterns` 함수 내에, RPC 호출 후 반환된 `ethics_refs`가 비어있을 경우 `logger.warning`을 통해 "규범을 찾을 수 없음 (is_active=false 가능성)" 로그를 명시적으로 남겨야 합니다.
    2. 장기적으로 `get_ethics_for_patterns` RPC가 `is_active=false`인 경우에도 어떤 정보를 반환할지(예: '비활성 관계' 플래그 포함) 정책을 재정의해야 합니다.

### **[축 2] 수평 정합성 (모듈 간 인터페이스)**

**판정: FAIL**

### **CRITICAL: `pipeline.py`의 치명적인 예외 처리 설계 오류**

- **문제점:** `pipeline.py`의 `analyze_article` 함수 83행을 보면, `CitationResolver`가 실패할 경우 `try/except` 블록으로 감싸고 로그만 남긴 채 **원본 리포트(`rr.report_text`)를 그대로 반환**합니다.
- **위험성:** 이 설계는 사용자에게 **`<cite>` 태그가 전혀 치환되지 않은, 미완성된 날것의 리포트**가 그대로 노출될 수 있음을 의미합니다. 이는 사용자 경험에 치명적이며, 시스템이 비전문적으로 보이게 만드는 심각한 결함입니다. 예외 처리는 시스템을 보호해야지, 망가진 결과를 사용자에게 보여줘서는 안 됩니다.
- **1차 감리가 놓친 이유:** 코드가 에러 없이 실행되는 'Happy Path'에 집중하면, 이처럼 예외 상황에서의 사용자 경험(UX) 문제를 놓치기 쉽습니다.
- **개선 제안 (필수):** `except` 블록에서 단순히 로그만 남기고 넘어갈 것이 아니라, `rr.report_text`를 **"리포트 후처리 중 오류가 발생했습니다. 잠시 후 다시 시도해주세요."** 와 같은 사용자 친화적인 오류 메시지로 대체해야 합니다.

### **MAJOR: `pattern_matcher.py`의 하드코딩된 모델명과 설정값**

- **문제점:** `pattern_matcher.py` 30행에 `HAIKU_MODEL = "claude-sonnet-4-6"`으로 하드코딩되어 있습니다. 변수명과 실제 값이 일치하지 않는 것은 물론, 모델명이나 `VECTOR_MATCH_COUNT` 같은 핵심 파라미터가 코드 내에 하드코딩되어 있어 변경이 매우 불편합니다.
- **위험성:** 배포 후 모델을 업그레이드하거나 파라미터를 튜닝할 때마다 코드를 직접 수정하고 재배포해야 합니다. 이는 유지보수성을 심각하게 저해하며, 운영 환경에서 실수를 유발할 가능성이 높습니다.
- **개선 제안:** 모든 모델명, `VECTOR_MATCH_COUNT`, `VECTOR_THRESHOLD` 등의 주요 설정값은 `.env` 파일로 분리하고, `os.environ.get()`을 통해 동적으로 로드하도록 코드를 전면 수정해야 합니다.

### **[축 3] 코드베이스 위생 (배포 준비 상태)**

**판정: PASS (단, 다수의 MAJOR/MINOR 이슈 존재)**

### **MAJOR: 레거시 모듈의 방치 (`analyzer.py`, `criteria_manager.py`, `prompt_builder.py`)**

- **문제점:** 감리 지침에 명시된 대로, `backend/core` 디렉토리에 현재 파이프라인에서 전혀 사용되지 않는 레거시 모듈 3개가 그대로 방치되어 있습니다.
- **위험성:**
    1. **신규 개발자의 혼란 가중:** 새로운 팀원이 합류했을 때, 어떤 코드가 활성 코드이고 어떤 코드가 레거시인지 파악하는 데 불필요한 시간을 낭비하게 됩니다.
    2. **잠재적 사이드 이펙트:** 누군가 실수로 이 레거시 모듈을 `import`하여 사용할 경우, 예기치 않은 버그가 발생할 수 있습니다.
- **개선 제안:** 이 3개의 모듈은 즉시 `backend/core/_deprecated/`와 같은 별도의 폴더로 옮기거나, 저장소에서 완전히 삭제하여 코드베이스를 정리해야 합니다.

### **MAJOR: `db.py`의 로컬 DB 우선 로직의 불안정성**

- **문제점:** `_get_supabase_config` 함수는 로컬 DB 연결을 확인하기 위해 실제 `httpx.get` 요청을 보냅니다. 이 로직은 타임아웃(5초)이 발생할 수 있으며, 로컬 개발 환경의 네트워크 상태에 따라 클라우드 DB로 잘못 폴백될 수 있는 불안정성을 내포하고 있습니다.
- **위험성:** 개발자가 로컬 DB로 테스트하려 했으나, 네트워크 문제로 자신도 모르게 클라우드 DB에 데이터를 쓰거나 변경하는 심각한 사고가 발생할 수 있습니다.
- **개선 제안:** `SUPABASE_LOCAL=1` 환경 변수를 **유일한 판단 기준으로 삼도록** 로직을 단순화해야 합니다. 환경 변수가 설정되어 있으면 무조건 로컬 DB 설정을 반환하고, 그렇지 않을 때만 클라우드 설정을 사용하도록 변경하여, 개발자의 의도를 명확하게 반영하고 예측 가능성을 높여야 합니다.

### **MINOR: 문서 및 설정 파일의 비일관성**

- `CLAUDE.md`: M4 시점에 멈춰 있어, M5의 최종 아키텍처(Sonnet Solo)와 플레이북 변경 사항이 전혀 반영되어 있지 않습니다.
- `pattern_matcher.py`: 파일 상단 주석에 여전히 "Haiku 호출"이라는 표현이 남아있어, 현재 Sonnet Solo 아키텍처와 일치하지 않습니다.
- **개선 제안:** 배포 전, 모든 문서와 주석을 최종 코드베이스 상태에 맞게 일괄적으로 업데이트하여 일관성을 확보해야 합니다.
