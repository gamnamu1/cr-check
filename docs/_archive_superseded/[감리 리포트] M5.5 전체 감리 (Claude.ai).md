## 축 1. 수직 정합성 — DB ↔ 코드 ↔ 프롬프트

### ✅ [PASS] RPC ↔ Python 컬럼 매핑

`search_pattern_candidates` RPC의 RETURNS TABLE 컬럼:
(pattern_id, pattern_code, pattern_name, pattern_description, similarity)
→ Python `search_vectors()`의 `row["pattern_code"]`, `row["similarity"]` 등 — **전부 정확히 일치**.

`get_ethics_for_patterns` RPC의 RETURNS TABLE 컬럼:
(pattern_id, pattern_code, ethics_code_id, ethics_code, ethics_title, ethics_full_text, ethics_tier, relation_type, strength, reasoning)
→ Python `fetch_ethics_for_patterns()`의 `row.get("ethics_code")` 등 — **전부 정확히 일치**.

SQL 내부의 CTE 별칭(p_code, ec_code 등)과 RETURNS TABLE 정의의 위치 매핑도 정확합니다.

### ✅ [PASS] cite 태그 형식 정합성

Sonnet 프롬프트(`report_generator.py`): `<cite ref="{ethics_code}"/>` 지시
→ CitationResolver 정규식: `<cite\s+ref="([^"]+)"\s*/>` — **매칭 확인**.
자기닫힘 변형(`</cite>`, 공백 포함)도 대응.

### ✅ [PASS] is_citable 필터 적용

`get_ethics_for_patterns` RPC의 직접 관계 CTE에 `AND ec.is_citable = TRUE` 조건 확인.
재귀 parent chain CTE에서도 `AND parent.is_citable = TRUE` 조건 확인.
is_citable=FALSE인 서문(JEC-P, JCE-P, JCP-P 등)이 인용 컨텍스트에서 제외됨.

### ✅ [PASS] 벡터 검색 파라미터 일관성

코드: `VECTOR_THRESHOLD = 0.2`, `VECTOR_MATCH_COUNT = 7`
마스터 플랜: "느슨한 threshold + Top-K"
세션 컨텍스트 v18: "threshold 0.2-0.25"
→ 일관됨.

### ⚠️ [MINOR] JEC/JCE 접두어 — 버그가 아닌 문서 명확성 문제

**발견**: D-01 E2E 테스트에서 `JEC-1: 진실을 추구한다`가 반환됨.
CLAUDE.md에는 "기자윤리강령 접두어는 JCE"라고 명시.
SESSION_CONTEXT에는 "KJA 절대 금지, JCE가 올바른 접두어"라고 명시.

**실제 DB 접두어 체계** (seed_data.sql에서 확인):

| 접두어 | 규범명 | 예시 |
|--------|--------|------|
| JEC | 언론윤리**헌**장 (Journalism **E**thics **C**harter) | JEC-1 "진실을 추구한다" |
| JCE | 기자윤리**강**령 (Journalists' **C**ode of **E**thics) | JCE-1 "언론자유 수호" |
| JCP | 기자윤리**실**천요강 | JCP-1-1 "언론자유 침해에 대한 단호한 대처" |
| PCE | 신문윤리강령 (Press Code of Ethics) | PCE-P 서문 |
| PCP | 신문윤리실천요강 | PCP-1-1 등 |

**결론**: JEC와 JCE는 **서로 다른 규범의 정상적인 접두어**이다. 버그가 아니다.
문서의 "JCE가 올바른 접두어"는 "KJA라는 잘못된 접두어를 JCE로 수정했다"는
역사적 맥락에서의 기술이며, "JEC라는 접두어가 잘못됐다"는 뜻이 아니다.
다만 문서만 읽으면 JEC가 오류처럼 보이므로 **문서 표기를 명확히 할 필요가 있다**.

**수정 권고**: CLAUDE.md와 SESSION_CONTEXT에서 접두어 관련 기술을 다음과 같이 갱신:
"KJA 접두어 절대 금지. DB에는 JEC(언론윤리헌장), JCE(기자윤리강령), JCP, PCE, PCP 등
 다양한 접두어가 존재하며, 모두 정상이다."

## 축 2. 수평 정합성 — 모듈 간 인터페이스

### ✅ [PASS] chunker → pattern_matcher 인터페이스

`chunker.py`: `chunk_article(text) → list[Chunk]`
`pipeline.py`: `chunk_texts = [c.text for c in chunks]` → `list[str]`로 변환
`pattern_matcher.py`: `match_patterns_solo(chunks: list[str], article_text: str, ...)`
→ 정확히 일치. Chunk.text(str)를 list[str]로 변환하여 전달.

### ✅ [PASS] pattern_matcher → report_generator 인터페이스

`pattern_matcher.py` 반환: `PatternMatchResult`
  - `validated_pattern_ids: list[int]` → `generate_report(pattern_ids: list[int])` ✅
  - `validated_pattern_codes: list[str]` → haiku_dicts 필터링에 사용 ✅
  - `haiku_detections: list[HaikuDetection]` → dict 변환 후 전달 ✅

pipeline.py에서의 dict 변환:
```python
haiku_dicts = [{"pattern_code": d.pattern_code, "matched_text": d.matched_text,
                "severity": d.severity, "reasoning": d.reasoning}
               for d in pm.haiku_detections
               if d.pattern_code in pm.validated_pattern_codes]
```
HaikuDetection의 4개 필드(pattern_code, matched_text, severity, reasoning)가
정확히 dict 키로 매핑됨 ✅

### ✅ [PASS] report_generator → citation_resolver 인터페이스

`report_generator.py` 반환: `ReportResult`
  - `report_text: str` — Sonnet 리포트 (cite 태그 포함)
  - `ethics_refs: list[EthicsReference]` — Sonnet에 전달된 규범 목록

`citation_resolver.py` 입력:
  `resolve_citations(report_text: str, ethics_refs: list[EthicsReference])`

EthicsReference 필드 매핑:
  - `ethics_code` → ref_map 키로 사용 ✅
  - `ethics_title` → 치환 형식 「{title}: ...」 ✅
  - `ethics_full_text` → 원문 발췌 ✅
  - `ethics_tier` → (가드레일 참고용, 현재 미사용이나 문제 없음) ✅

`get_ethics_for_patterns` RPC 반환 컬럼명 `ethics_code` ↔
`fetch_ethics_for_patterns()`의 `row.get("ethics_code")` ↔
`EthicsReference.ethics_code` ↔ `resolve_citations()`의 `ref_map` 키
→ **4단계 매핑이 모두 정확히 일치** ✅

### ✅ [PASS] deprecated 코드 간섭 위험

`pipeline.py`의 import:
```python
from .pattern_matcher import (
    match_patterns_solo,
    match_patterns_2call,  # deprecated 2-Call (비교용)
    match_patterns,        # deprecated 1-Call (비교용)
    PatternMatchResult,
)
```
import는 존재하나, `analyze_article()` 함수 내에서 실제 호출은
**`match_patterns_solo()`만** 사용. deprecated 함수는 호출되지 않음 ✅

다만, import 자체가 모듈 로드 시 불필요한 코드를 실행시키진 않으므로
(함수 정의만 로드), 런타임 간섭 위험은 없음.

### ✅ [PASS] 벤치마크 스크립트 아키텍처 정합성

`benchmark_pipeline_v3.py`:
  - `from core.pipeline import analyze_article` → 현재 활성 pipeline 사용 ✅
  - `result = analyze_article(article_text, run_sonnet=False)` → 패턴 식별까지만 실행 ✅
  - `pm.suspect_result` 참조 → `SuspectResult` (Solo 모드에서 호환용 빈 값 제공) ✅
  - Solo 모드에서 `suspect_categories`는 항상 `[]` → 스크립트가 이를 올바르게 처리 ✅

### ⚠️ [MAJOR] main.py가 구 MVP analyzer를 사용

```python
# backend/main.py
from core.analyzer import ArticleAnalyzer  # ← 구 MVP 모듈
```

`main.py`(FastAPI 엔트리포인트)는 M4에서 구축한 `pipeline.py`를 사용하지 않고,
M1 이전의 `core/analyzer.py`를 사용하고 있다. 즉:

- 벤치마크 스크립트: `pipeline.py` (Sonnet Solo + CitationResolver) ← **현재 아키텍처**
- 웹 API: `analyzer.py` (구 MVP, 정적 JSON 기반) ← **구 아키텍처**

**영향**: M6에서 클라우드 배포 시 `main.py`를 `pipeline.py` 기반으로 교체해야 함.
현재 벤치마크 결과는 정확하지만, 실제 서비스는 다른 코드를 실행하게 됨.

**수정 시점**: M6 Phase 1 아카이빙 통합 시. 현시점에서는 인지하고 넘어감.

### ✅ [PASS] 에러 핸들링

pipeline.py: CitationResolver를 `try/except`로 감싸 graceful fallback ✅
pattern_matcher.py: `_parse_solo_response()`에서 JSON 파싱 실패 시 빈 결과 반환 ✅
report_generator.py: `pattern_ids` 빈 리스트 시 조기 반환 ✅
citation_resolver.py: 빈 report_text 시 조기 반환, 환각 ref 제거 + 로그 ✅
