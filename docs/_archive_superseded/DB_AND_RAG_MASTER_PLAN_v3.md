### 0.3 핵심 갭(Gap) — 이전 문서에서 누락된 사항

이전 4개 문서는 모두 **"분석 결과 저장 + 사용자 관리 + 통계"**에 집중하고 있으며, CR-Check의 가장 근본적인 기술 부채인 **"119개 보도관행 패턴 ↔ 언론윤리규범의 정적 하드코딩 매핑"** 문제를 다루지 않았다.

이 문제를 해소하지 않으면 DB에 아무리 좋은 결과를 저장해도, 그 결과 자체의 **정확성과 근거**가 취약한 상태로 남는다. 따라서 본 문서는 DB 구축과 RAG 도입을 하나의 통합 계획으로 다룬다.

## 1. 확장된 로드맵 — 5 Phase 구조

이전 문서의 Phase 1-3 구조를 유지하되, **Phase 0(RAG 지식베이스)**을 선행 삽입하고 **Phase 4(커뮤니티)**를 추가한다.

```
Phase 0: RAG 지식베이스 구축 ← [신규] 핵심 추가
    ↓
Phase 1: 아카이빙 (분석 결과 저장)  ← 기존 유지
    ↓
Phase 2: 사용자 인증 및 개인화     ← 기존 유지
    ↓
Phase 3: 통계 대시보드              ← 기존 유지
    ↓
Phase 4: 커뮤니티 (북마크, 피드백)  ← 기존 FINAL의 Phase 4+
```

### Phase 0: RAG 지식베이스 구축 [신규] — 3-5일

**핵심 가치**: "분석의 근거를 동적으로 확보한다"

**해결하는 문제**:
- 119개 패턴 ↔ 윤리규범의 정적 JSON 매핑 → 동적 검색+추론으로 대체
- AI 윤리 코드 인용 환각 → 실제 규범 텍스트 기반 grounding으로 방지
- 매핑 근거 부재 → 관계 테이블의 reasoning 필드로 설명가능성 확보

**기술 선택: Hybrid RAG (Vector Search + Relational Graph)**
- Vector Search: Supabase pgvector (기사 텍스트 ↔ 패턴/규범 의미 매칭)
- Relational Graph: Supabase 관계형 테이블 (패턴→규범 관계 + 근거)
- 이유: 전용 Graph DB(Neo4j 등) 없이 Supabase 단일 플랫폼으로 통합 가능

**작업**:
1. pgvector 확장 활성화
2. `patterns`, `ethics_codes`, `pattern_ethics_relations` 테이블 생성
3. 119개 패턴 + 윤리규범 텍스트 임베딩 생성 및 저장
4. 관계 데이터 초기 구축 (기존 JSON 매핑을 시드 데이터로 활용)
5. 하이브리드 검색 함수 구현 (벡터 유사도 → 관계 테이블 JOIN)

**블로킹 요소**: 임베딩 모델 선정 (Voyage AI vs OpenAI vs 오픈소스)

## 2. 통합 데이터베이스 스키마

### 2.1 Phase 0: RAG 지식베이스 테이블 [신규]

```sql
-- ============================================
-- CR-Check RAG 지식베이스 스키마
-- Phase 0: 정적 매핑 → 동적 RAG 전환
-- ============================================

-- pgvector 활성화
CREATE EXTENSION IF NOT EXISTS vector;

-- 1. 문제적 보도관행 패턴 (119개)
CREATE TABLE public.patterns (
  id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  code TEXT NOT NULL UNIQUE,        -- 예: "1-1-1", "2-3-2"
  name TEXT NOT NULL,               -- 패턴명
  description TEXT NOT NULL,        -- 상세 설명
  category TEXT,                    -- 상위 분류 (예: "취재윤리")
  subcategory TEXT,                 -- 하위 분류
  description_embedding vector(1536), -- 임베딩 벡터
  created_at TIMESTAMPTZ DEFAULT now(),
  updated_at TIMESTAMPTZ DEFAULT now()
);

CREATE INDEX idx_patterns_code ON public.patterns(code);
CREATE INDEX idx_patterns_embedding ON public.patterns
  USING ivfflat (description_embedding vector_cosine_ops) WITH (lists = 10);

-- 2. 언론윤리규범 조항
CREATE TABLE public.ethics_codes (
  id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  code TEXT NOT NULL UNIQUE,        -- 예: "KJA-3", "PCC-7.2"
  title TEXT NOT NULL,              -- 조항 제목
  full_text TEXT NOT NULL,          -- 조항 전문
  source TEXT NOT NULL,             -- 출처 (한국기자협회, 방송통신심의위 등)
  article_number TEXT,              -- 조항 번호
  text_embedding vector(1536),     -- 임베딩 벡터
  created_at TIMESTAMPTZ DEFAULT now(),
  updated_at TIMESTAMPTZ DEFAULT now()
);

CREATE INDEX idx_ethics_code ON public.ethics_codes(code);
CREATE INDEX idx_ethics_embedding ON public.ethics_codes
  USING ivfflat (text_embedding vector_cosine_ops) WITH (lists = 10);

-- 3. 패턴 ↔ 윤리규범 관계 (간이 지식 그래프)
-- ★ 이 테이블이 정적 JSON 매핑을 대체하는 핵심
CREATE TABLE public.pattern_ethics_relations (
  id BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  pattern_id BIGINT REFERENCES public.patterns(id) ON DELETE CASCADE NOT NULL,
  ethics_code_id BIGINT REFERENCES public.ethics_codes(id) ON DELETE CASCADE NOT NULL,
  relation_type TEXT NOT NULL CHECK (relation_type IN (
    'violates',      -- 직접 위반
    'related_to',    -- 간접 관련
    'exception_of'   -- 특정 조건에서 예외
  )),
  strength TEXT DEFAULT 'moderate' CHECK (strength IN ('strong', 'moderate', 'weak')),
  reasoning TEXT,       -- 왜 이 관계인지 설명 (설명가능성 핵심)
  examples TEXT,        -- 실제 사례 참조
  created_at TIMESTAMPTZ DEFAULT now(),

  CONSTRAINT unique_pattern_ethics UNIQUE (pattern_id, ethics_code_id, relation_type)
);

CREATE INDEX idx_rel_pattern ON public.pattern_ethics_relations(pattern_id);
CREATE INDEX idx_rel_ethics ON public.pattern_ethics_relations(ethics_code_id);
```

```sql
-- 4. 하이브리드 검색 함수 (Vector + Relational)
-- 기사 텍스트로부터 관련 패턴 + 윤리규범을 동적으로 검색
CREATE OR REPLACE FUNCTION search_patterns_with_ethics(
  query_embedding vector(1536),
  match_threshold FLOAT DEFAULT 0.7,
  match_count INT DEFAULT 10
)
RETURNS TABLE (
  pattern_id BIGINT,
  pattern_code TEXT,
  pattern_name TEXT,
  pattern_description TEXT,
  similarity FLOAT,
  related_ethics JSONB  -- [{code, title, relation_type, reasoning}]
) AS $$
BEGIN
  RETURN QUERY
  SELECT
    p.id as pattern_id,
    p.code as pattern_code,
    p.name as pattern_name,
    p.description as pattern_description,
    1 - (p.description_embedding <=> query_embedding) as similarity,
    COALESCE(
      jsonb_agg(
        jsonb_build_object(
          'code', ec.code,
          'title', ec.title,
          'relation_type', per.relation_type,
          'strength', per.strength,
          'reasoning', per.reasoning
        )
      ) FILTER (WHERE ec.id IS NOT NULL),
      '[]'::jsonb
    ) as related_ethics
  FROM public.patterns p
  LEFT JOIN public.pattern_ethics_relations per ON p.id = per.pattern_id
  LEFT JOIN public.ethics_codes ec ON per.ethics_code_id = ec.id
  WHERE 1 - (p.description_embedding <=> query_embedding) > match_threshold
  GROUP BY p.id, p.code, p.name, p.description, p.description_embedding, query_embedding
  ORDER BY similarity DESC
  LIMIT match_count;
END;
$$ LANGUAGE plpgsql STABLE;

COMMENT ON FUNCTION search_patterns_with_ethics IS 
  '하이브리드 RAG: 벡터 유사도로 패턴 후보 검색 → 관계 테이블 JOIN으로 윤리규범 매핑';
```

### 3.1 현재 구조 (As-Is)

```

```

### 3.2 RAG 통합 후 구조 (To-Be)

```
기사 URL → 스크래핑 → 임베딩 생성
                          ↓
                   search_patterns_with_ethics()
                   (pgvector 유사도 → 관계 테이블 JOIN)
                          ↓
                   동적 RAG 컨텍스트 생성
                   (관련 패턴 + 윤리규범 + 매핑 근거)
                          ↓
              Claude Haiku (패턴 식별) ← RAG 컨텍스트 주입
                          ↓
              Claude Sonnet (보고서) ← RAG 컨텍스트 + 근거 주입
                          ↓
              분석 결과 → Supabase 저장 (analysis_results)
```

### 3.3 핵심 변경점

| 항목 | As-Is | To-Be |
|------|-------|-------|
| 패턴-규범 매핑 | 코드 내 정적 JSON | DB 관계 테이블 (동적 검색) |
| 규범 인용 | 하드코딩된 텍스트 | 실제 규범 원문 기반 (grounding) |

## 5. 실행 순서 및 타임라인

### Phase 0과 Phase 1의 실행 순서 결정

**옵션 A: Phase 0 → Phase 1 (RAG 먼저)**
- 장점: 처음부터 RAG 기반 분석 결과가 저장됨
- 단점: 데이터 축적 시작이 늦어짐

**옵션 B: Phase 1 → Phase 0 (저장 먼저)** ← 권장
- 장점: 즉시 데이터 축적 시작, RAG는 점진적 도입
- 단점: 초기 저장 데이터는 정적 매핑 기반

**옵션 C: Phase 0 + Phase 1 동시 진행** ← 실현 가능하면 최적
- Supabase 프로젝트 생성 시 RAG 테이블 + 기본 테이블 모두 생성
- 백엔드에서 RAG 검색과 결과 저장을 동시에 구현

→ **권장: 옵션 C 또는 B**. Supabase 셋업 시 모든 테이블을 한 번에 생성하되, RAG 검색 로직은 Phase 1 배포 후 점진적으로 연결.

### 통합 타임라인

