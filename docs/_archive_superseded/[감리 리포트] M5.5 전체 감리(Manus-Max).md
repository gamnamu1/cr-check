## 축 1: 수직 정합성 감리 (DB ↔ 코드 ↔ 프롬프트)

### PASS 항목

DB 스키마(`create_cr_check_schema.sql`)에 정의된 `search_pattern_candidates` RPC와 `get_ethics_for_patterns` RPC가 파이썬 코드(`pattern_matcher.py`, `report_generator.py`)에서 정확한 매개변수 타입과 구조로 호출되고 있습니다. 특히 `get_ethics_for_patterns`의 재귀 CTE(parent chain rollup)가 `citation_resolver.py`의 in-memory 매칭 전략과 올바르게 맞물려, Sonnet이 제공받은 규범 컨텍스트 내에서만 인용이 이루어지는 설계 의도가 코드 레벨에서 충실히 구현되어 있습니다.

`seed_data.sql`과 `data_implant_pattern_desc.sql`의 패턴 데이터가 `pattern_matcher.py`의 `_load_pattern_catalog()` 함수에서 조회하는 조건(`hierarchy_level=3`, `is_meta_pattern=false`)과 정확히 일치합니다.

### 경미 사항: 모델 버전 지정의 혼란

`pattern_matcher.py` 36라인에서 `HAIKU_MODEL = "claude-sonnet-4-6"`으로 되어 있어, 변수명(`HAIKU_MODEL`)과 실제 모델(`sonnet`)이 불일치합니다. 이는 M5에서 Sonnet Solo로 전환하면서 변수명을 갱신하지 않은 잔재로 보입니다. 한편, `report_generator.py`에서는 `SONNET_MODEL = "claude-sonnet-4-20250514"`로 별도의 모델 버전을 참조하고 있어, 동일한 Sonnet 모델을 사용하는 두 모듈이 서로 다른 버전 문자열을 가지고 있습니다.

> **권장:** 모델 식별자를 환경변수(`ANTHROPIC_MODEL`)로 통합 관리하거나, 최소한 변수명을 `PRIMARY_MODEL` 등으로 통일하여 혼란을 제거하는 것이 좋습니다.

## 축 2: 수평 정합성 감리 (모듈 간 인터페이스)

### PASS 항목

`pipeline.py`의 `analyze_article()` 함수가 M5 모듈들을 올바른 순서로 오케스트레이션하고 있습니다.

1. `chunker.chunk_article()` → 청크 리스트 생성
2. `pattern_matcher.match_patterns_solo()` → Sonnet Solo 패턴 식별
3. `report_generator.generate_report()` → 규범 조회 + Sonnet 리포트 생성
4. `citation_resolver.resolve_citations()` → 결정론적 인용 후처리

각 모듈 간 데이터 타입(`PatternMatchResult`, `ReportResult`, `EthicsReference`)이 일관되게 사용되고 있으며, `citation_resolver`가 실패하더라도 원본 리포트가 유지되도록 `try-except` 방어 로직이 적용된 점이 훌륭합니다.

`pipeline.py`에서 `match_patterns_2call`과 `match_patterns`를 `# deprecated 비교용`으로 import하고 있는 것은 벤치마크 비교 목적으로 이해되며, 실제 실행 경로에서는 `match_patterns_solo`만 호출되므로 기능적 문제는 없습니다.

## 축 3: 코드베이스 위생 감리 (M5 작업물 한정)

### 권장 사항 1: `pattern_matcher.py` 내부의 레거시 코드 정리

`pattern_matcher.py`는 총 1,337라인 중 약 830라인(508~1,338라인)이 `[DEPRECATED]` 주석이 달린 과거 2-Call 및 1-Call 로직입니다. M5의 활성 코드(Sonnet Solo)는 약 500라인에 불과합니다.

| 구간 | 라인 수 | 상태 |
| --- | --- | --- |
| 공통 인프라 + Sonnet Solo (1~506) | ~506 | ✅ 활성 |
| 2-Call 아키텍처 (508~966) | ~458 | ❌ DEPRECATED |
| 1-Call Haiku (967~1,338) | ~371 | ❌ DEPRECATED |

벤치마크 비교 목적으로 남겨둔 것으로 이해되지만, M6 배포 시에는 과감히 삭제하는 것을 권장합니다. Git 히스토리가 코드를 기억하고 있으므로, 필요 시 언제든 복원할 수 있습니다. 파일 크기를 1/3로 줄이면 향후 유지보수 시 인지적 과부하가 크게 감소합니다.

### 권장 사항 2: `requirements.txt` 의존성 누락

M5 핵심 모듈이 실제로 사용하는 패키지 중 `requirements.txt`에 누락된 항목이 있습니다.

| 패키지 | 사용처 | `requirements.txt` 포함 여부 |
| --- | --- | --- |
| `openai` | `pattern_matcher.py` (임베딩 생성) | ❌ 누락 |
| `httpx` | `db.py`, `pattern_matcher.py`, `report_generator.py` | ❌ 누락 |

현재 `anthropic` 패키지가 `httpx`를 전이 의존성(transitive dependency)으로 설치해주기 때문에 우연히 작동하고 있지만, 이는 명시적이지 않아 향후 `anthropic` 버전 업데이트 시 깨질 수 있습니다. `openai`와 `httpx`를 `requirements.txt`에 명시적으로 추가해야 합니다.

### 권장 사항 3: `db.py`의 하드코딩된 로컬 JWT

`db.py` 16~18라인에 Supabase 로컬 테스트용 Service Role Key(JWT)가 하드코딩되어 있습니다. 이 키는 Supabase 공식 로컬 개발용 기본 키이므로 보안 사고로 이어지지는 않지만, 소스 코드에 토큰을 하드코딩하는 패턴은 코드 리뷰 도구나 보안 스캐너에서 경고를 발생시킬 수 있습니다. 환경변수(`SUPABASE_LOCAL_KEY`)로 분리하는 것이 깔끔합니다.

## 최종 소견

**M5 작업물 자체의 품질은 매우 우수합니다.** DB 스키마 설계부터 RPC 함수, 벡터 검색, Sonnet Solo 프롬프트, 결정론적 인용 후처리까지 — 각 모듈이 명확한 책임을 가지고 깔끔하게 연결되어 있습니다. 특히 `pipeline.py`의 오케스트레이션은 간결하면서도 방어적이어서, 프로덕션 환경에서의 안정성을 기대할 수 있습니다.

M6 통합 배포의 핵심 과제는 M5의 우수한 파이프라인을 기존 서빙 레이어(`main.py` → 프론트엔드)에 올바르게 연결하는 "배관 작업(Plumbing)"입니다. 위의 통합 과제 체크리스트(T-1 ~ T-8)를 순서대로 수행하시면 성공적인 배포가 가능할 것입니다.
