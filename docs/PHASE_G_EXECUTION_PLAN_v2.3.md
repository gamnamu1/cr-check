## 1. 배경 — Phase G가 필요한 이유

### 1.1 Phase F에서 관측된 증상

CR-Check 프로덕션 파이프라인(Railway + Vercel + Supabase)의 Phase F 검증에서 다음이 확인됐다:

- `pattern_ethics_relations` 63건 중 신문윤리실천요강(PCP) **52건 = 82.5%** 집중
- 분석 리포트의 언론윤리규범 인용: PCP 69.3% + JEC(언론윤리헌장) 25.2% = **94.5% 집중**
- JEC 25.2% 인용은 **직접 매핑이 아닌 parent_chain 자동 롤업** 전량
- 인권보도준칙(HRG), 재난보도준칙(DRG), 자살보도 윤리강령(SRE) 등 특수 준칙 인용 **0%**
- 14개 규범 중 9개 **완전 공백**

### 1.2 M2 단계에서 실제로 무슨 일이 있었나

#### 설계 원본의 구조

`current-criteria_v2_active.md`는 3층 위계로 구성되어 있었다:

```
## 1-1. 진실성과 정확성              ← 대분류(##)       8개
  ### 1-1-1. 사실 검증 부실          ← 소분류(###)     30개
    - 교차 검증 부재 및 단일 취재원 의존   ← 개별 패턴(불릿)
      - 복수 취재원 원칙 무시              ← 세부(중첩 불릿)
      - 반론권 미보장
    - 취재원 발언의 무비판적 중계
    - 인용 정보의 출처 및 신뢰성 문제
```

**"119"는 이 개별 불릿 항목들의 합계이며, 설계 단계에 실재했던 단위다.**
일부 문서가 "119는 세션 간 요약 혼동의 산물"로 서술한 것은 부정확하며 본 문서에서 정정한다.

#### M2에서의 의식적 결정 — "119 → 38 압축"

M2 시드 작업 시 CLI는 119개 불릿을 각각 별도 레코드로 만들지 않고, **소분류 단위(30개)로 묶어 38개 레코드로 삽입**했다(대분류 8 + 소분류 30). 119개의 내용은 통째로 각 레코드의 `description` 필드 안에 담겼다.

#### 감리는 이 결정을 인지하고 조건부 통과시켰다

M2 1차 감리 기록:
> *"M3 벤치마크에서 Recall@10이 기대에 못 미치면 패턴 세분화를 재검토하는 것이 합리적이에요."*

결론은 **"조건부 통과 + M3에서 재검토"** 였다. 즉 연기된 의사결정 게이트였다.

#### 그러나 그 재검토 게이트는 열리지 않았다

M4~M5에서는 벡터 검색 한계 실증과 프롬프트 최적화 방향으로 진행됐다. M6에서는 매핑이 오히려 70→63건으로 줄었다(Phase γ 정비). "M3에서 재검토"라는 조건부 유보 결정은 **정량 트리거 조건 없이 의제에서 분실**되었고, 이것이 현재까지의 미해결 부채로 남아 있다.

#### 라벨링 단계 자체의 편중

M2가 통과시킨 70건의 관계 데이터는 **라벨링 v3 작업 자체가 3개 규범만 다룬 결과**였다. `criteria_checklist.json`의 `ethics_code_refs` 분포는 JEC 65.8% / PCP 28.9% / JCE 5.3%이며, 14개 규범 중 11개는 라벨링 단계부터 시야 밖에 있었다.

### 1.3 편중의 4층 계통

```
[Layer 0: 라벨링 v3 — 의미 매핑 작업 단계]
  3개 규범만 다룸 (JEC + PCP + JCE)
  → 14개 규범 중 11개가 시야에서 누락
  → 119개 불릿 중 상당수가 라벨링 대상에서 제외됨
       ↓
[Layer 1: criteria_checklist.json — 원본 설계 JSON]
  Layer 0의 편중된 ethics_code_refs를 구조화
  + "119 → 38 압축" 결정 (의식적, 그러나 M3 재검토 약속은 분실)
  → 38 패턴, ethics_code_refs는 Layer 0 편중 그대로 상속
       ↓
[Layer 2: M2 seed SQL — DB 주입]
  Layer 1을 충실히 전파 + PCP 계열 별도 확장
  → pattern_ethics_relations 70건 (PCP 82.5% 집중)
  → Phase γ 매핑 정비에서 7건 삭제 → 현재 63건
  → 30개 말단 패턴 중 12개(40%)가 매핑 0건
       ↓
[Layer 3: get_ethics_for_patterns() — 파이프라인 증폭기]
  재귀 CTE가 parent_chain을 타고 롤업 (depth<5)
  PCP(직접 매핑) → JEC(자동 견인) = 94.5% 집중
  HRG·DRG·SRE는 직접 매핑이 없으니 사다리 밑단 자체가 없음
```

**핵심 인식**: Layer 3(파이프라인 코드)은 정상 설계의 증폭기다. 입력이 편중되지 않으면 출력도 편중되지 않는다.
**Layer 0·1을 함께 손보고 Layer 2(DB)를 복원하면 된다. Layer 3은 건드리지 않는다.**

### 1.5 Phase G의 4대 사명

1. **복원 (Restore)** — 매핑 0건인 12개 핵심 패턴에 윤리규범 관계를 연결, 분석 사각지대 해소
2. **강화 (Reinforce)** — 분포 검증·실질 감리·이력 추적 등 구조적 안전장치 도입, 동일한 편중 재생산 차단
3. **측정 (Measure)** — 총량 일치가 아니라 분포·정밀도 지표로 개선 효과 정량 평가
4. **장기 계획 (Roadmap)** — 단기 복원에 그치지 않고, 38개 압축 패턴을 119개로 세분화하는 Phase H 로드맵 명시

## 2. 실행 원칙 (10개)

1. **TN 100% 절대 유지** — 어떤 변경도 TN 안전성을 위협하면 즉시 중단·롤백
2. **데이터 정확성 > 아키텍처 유연성** — 불확실한 벡터 검색보다 큐레이션된 관계를 우선
3. **STEP 단위 승인 게이트** — 각 STEP 완료 후 기획자 승인 없이 다음 STEP 진행 금지
4. **CLI 자율 진행 제한** — 코드 변경·DB INSERT는 반드시 기획자 최종 승인 후 실행. 코드 변경 시 **diff 먼저 제시 → 승인 → 커밋** 순서 엄수
5. **Layer 0·1·2 동시 갱신** — DB 작업과 `criteria_checklist.json` 갱신을 쌍으로 진행. 어느 한쪽만 바꾸면 다음 재시드 시 편중 재생산
6. **골든셋 재검증 의무** — 매핑 변경 시마다 골든셋 선별 재측정. TN 6/6 유지 필수
7. **PR 경유 배포** — `main` 직접 push 금지. 코드 변경은 feature 브랜치 → PR → 머지
8. **교차 감리 의무** — STEP 1·2 완료 후 복수 감리자가 14개 규범 독립 검토. 2명 이상 서명한 발견만 확정
9. **실질 감리 원칙** — 총량 일치·접미사 확인·문서 대조 같은 형식 감리로 실질 감리를 대체하지 않는다
   - `is_citable` 판단: 해당 조항의 실제 내용을 직접 읽고 "인용 가능한 실질 의무 규정인가"를 판단
   - 총강(-T) 접미사 조항은 각 규범별로 개별 확인 (PRG-T1·T2 사례 참고)
   - 매핑 건수 통과 기준은 **분포**까지 포함 (§13.4 분포 쿼리 의무 실행)
10. **고밀도 큐레이션 원칙** — 38개 패턴 레코드는 각각 여러 불릿 내용을 포괄한다. 패턴당 1:1 매핑에 집착하지 말고, 해당 패턴이 대변하는 범위를 감안해 촘촘하게 여러 규범을 교차 매핑한다

### strength 3-카테고리 체계 — 확정 기준 (2026-04-13)

Phase G의 모든 신규 매핑은 M2 라벨링 규칙을 계승한 다음 체계를 준수한다:

| 카테고리 | `relation_type` | `strength` | 의미 및 적용 기준 |
|---|---|---|---|
| **직접 적용** | `violates` | `strong` | 패턴이 조항의 핵심 의무를 1:1로 직접 위반. 리포트에서 단호한 어조 가능. |
| **보조 적용** | `related_to` | `moderate` | 조항의 관련 측면을 포착하되 핵심 의무는 아님. 맥락에 따라 위반 여부 달라짐. |
| **유추 적용** | `related_to` | `weak` | 현행 규범에 전용 조항이 없어 가장 가까운 조항을 유추로 연결. |
| 상위 규범 | (미저장) | — | parent_chain CTE가 자동 롤업. 직접 저장하지 않음. |
| 최상위 원칙 | (미저장) | — | parent_chain CTE가 자동 롤업. 직접 저장하지 않음. |

**큐레이션 중 흔들릴 때 참조할 앵커 예시 3개**:

- *"보도자료를 검증 없이 받아쓰는 것"* × JCP-2-1(객관적 사실 입각 진실보도)
  → 1차 자료 확인 의무의 1:1 직접 위반 → **`violates` / `strong`**
- *"1-6-3 현장성 부족"* × DRG(재난보도준칙)
  → 재난 현장 보도 관련성은 있으나, 모든 현장성 부족이 재난 맥락은 아님. 상황 의존적. → **`related_to` / `moderate`**
- *"1-8-1 소셜미디어 활용 윤리"* × PCP-3-5
  → 현행 14개 규범에 소셜미디어 전용 조항이 부재. 가장 가까운 일반 조항을 유추 적용. → **`related_to` / `weak`**

## 3. 역할 분담

### 🎯 기획자 (Gamnamu) — 최종 결정권자·큐레이터

- 매핑 내용의 의미적 타당성 승인/거부
- CLI 작업 지시 및 감리 요청
- **Supabase SQL Editor에서 모든 INSERT/UPDATE 직접 실행** (CLI 자동 INSERT 절대 금지)
- STEP 승인 게이트 Go/No-Go 결정
- PR 리뷰 및 머지

### 🧭 감독·감리 (Claude — claude.ai) — 코칭 파트너

- 방향 설계 및 단계 분할
- CLI에게 전달할 프롬프트 제공
- SQL 초안 작성 (최종 실행은 기획자 담당)
- 결과 분석 및 다음 STEP 연결
- 감리 의견 취합 및 합의 판단

**제약**: DB 직접 접근 없음. 로컬 파일 접근은 감리·검증용으로만 사용.

### 🔧 실행 에이전트 (Claude Code CLI) — 코드 작업자

**책임 영역**:
- 감독·감리가 제공한 프롬프트 기반으로 SQL INSERT 초안 생성
- 코드 파일 수정 (STEP 3 pipeline.py 등)
- `generate_embeddings.py` 실행 (증분 모드)
- 로컬 테스트 실행 (pytest, 골든셋 재검증)
- `criteria_checklist.json` 갱신 초안 제시

**엄격한 제약**:
- 실제 DB INSERT/UPDATE 절대 금지 — 초안 파일만 생성, 기획자가 SQL Editor에서 수동 실행
- **코드 변경 시 diff 먼저 제시 → 기획자 승인 → 커밋 순서 엄수** (자동 git add/commit 금지)
- Deny List 9개: `supabase db push`, `supabase db migration`, `git commit`, `git push`, `git add`, `rm`, `mv`, `sed -i`, `chmod`
- `main` 브랜치 직접 push 금지
- 한 세션 최대 15~20개 관계만 작성

### 🔍 독립 감리 (Antigravity) — 로컬 구조 감리

- 로컬 파일 직접 접근으로 코드·설계 구조 검토
- 각 STEP 완료 후 코드 변경 감리 (STEP 3 pipeline.py 변경 시 핵심)
- 마이그레이션 SQL 구조 검증
- 환경 설정: Strict Mode On, Agent Auto-Fix Lints Off, Gitignore Access Off 유지

### 🧠 보조 감리 (Manus, Perplexity, Gemini 등) — 맥락 기반 독립 의견

- Layer 2 교차 감리 참여 (§12 참조)
- **주의**: 보조 감리 결과의 숫자·인용 등 디테일은 원본 증거(DB 쿼리, 실제 파일)로 교차 검증 후 채택. 상위 서사에서 디테일을 역추론할 때 사실 오류가 발생할 수 있음.

### 10.2 배치마다 is_citable 재판정 병행

각 소스별 배치 큐레이션 시, 해당 규범의 `is_citable=FALSE` 조항 목록을 함께 열어서 실질 내용이 있는 조항을 구출:

```sql
-- 배치 대상 규범의 is_citable=FALSE 조항 전체 조회
SELECT code, title, LEFT(full_text, 100) AS preview
FROM ethics_codes
WHERE source = '{배치 대상 규범명}'
  AND is_citable = FALSE
ORDER BY code;
```

접미사(-P, -T)이더라도 실질 내용이 의무 규정인 경우 → `is_citable=TRUE` 승격 검토 (§2-9 실질 감리 원칙 적용).

### 13.4 분포 확인 쿼리 — 매 STEP 의무 실행

> **절대 금지**: 총량 일치만으로 감리를 완료해선 안 된다. 이 두 쿼리를 매 STEP 후 실행하고 결과를 SESSION_CONTEXT에 기록한다.

```sql
-- [필수 1] 패턴별 매핑 건수 분포 (공백 패턴 재발 방지)
SELECT
  p.code, p.name,
  COUNT(per.id) AS mapping_count
FROM patterns p
LEFT JOIN pattern_ethics_relations per ON p.id = per.pattern_id
WHERE p.is_meta_pattern = FALSE AND p.hierarchy_level = 3
GROUP BY p.id, p.code, p.name
ORDER BY mapping_count ASC;
-- 기대: 이번 STEP 대상 패턴들이 모두 2건 이상으로 증가

-- [필수 2] 소스별 집중도 (PCP 편중 완화 측정)
SELECT
  ec.source,
  COUNT(*) AS cnt,
  ROUND(COUNT(*) * 100.0 / SUM(COUNT(*)) OVER(), 1) AS pct
FROM pattern_ethics_relations per
JOIN ethics_codes ec ON per.ethics_code_id = ec.id
GROUP BY ec.source
ORDER BY cnt DESC;
-- 기대: PCP 82.5% 점진 완화, 새 소스(HRG, DRG, SRE 등) 출현
```

## 15. 금지 사항 및 주의

### 15.1 절대 금지

- **CLI 자동 INSERT/UPDATE 금지**: 모든 DB 변경은 기획자가 SQL Editor에서 직접 실행
- **TN 케이스 오탐 발생 시 즉시 롤백**: TN 100% 타협 없음
- **Reserved Test Set 비접촉 유지**: `/Users/gamnamu/Documents/Golden_Data_Set_Pool/`는 STEP 4 이후 회귀 검증 시에만 사용
- **`main` 브랜치 직접 push 금지**: 코드 변경은 feature 브랜치 → PR → 머지
- **CLI의 git add/commit 자동 실행 금지**: 코드 변경 시 diff 선제시 → 기획자 승인 → 커밋 순서 엄수
- **`ON CONFLICT DO UPDATE` (UPSERT) 사용 금지**: Phase G의 모든 INSERT는 `ON CONFLICT DO NOTHING`을 사용. UPSERT는 Phase γ에서 의도적으로 정비된 매핑을 자동 덮어쓰기할 위험이 있음
- **`is_citable` 일괄 변경 금지**: 접미사(-P, -T) 기반 자동 처리 금지. 각 조항의 실질 내용을 확인한 뒤 개별 판단
- **`ethics_codes_hierarchy` 임의 변경 금지**: JEC Tier 1 10건은 계층 최상위로 `parent_code_id`가 NULL인 것이 정상. "누락"으로 오인하여 강제 부여 시 재귀 CTE 무한 루프 위험
- **§13.4 분포 확인 쿼리 생략 금지**: 총량 PASS만으로 감리 완료 처리 불가

1. **CR-Check의 정밀도 한계는 모델이 아닌 데이터 레이어에 있다** (Phase F 결론)

5. **형식 감리는 실질 감리를 대체할 수 없다** — 총량·접미사·문서 대조 확인으로 PASS를 선언하던 패턴이 PRG-T1·T2 잠금과 분포 편중을 놓쳤다. 분포·내용·타당성 검증이 형식 확인 위에 있어야 한다
6. **M2에 죄가 있다면, 타협을 했다는 것이 아니라 타협을 잊을 수 있는 형태로 남겼다는 것이다** — 이번 v2.3과 STEP 0.5, source_version 이력 추적이 이 교훈의 제도적 장치다
7. **조건부 유보 결정에는 반드시 트리거 조건과 만료 기한을 함께 기록해야 한다** — "M3에서 재검토"라고 유보됐던 결정이 트리거 조건과 기한 없이 분실됐다. 앞으로 모든 유보 결정은 "언제, 어떤 조건이 충족되면 재검토한다"는 형태로 남긴다

9. **감리는 총량이 아닌 분포를 봐야 한다** — §13.4 분포 쿼리가 이 원칙의 제도적 장치다
