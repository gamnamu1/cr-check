# 오류 진단 및 해법_제미나이

이번 에러는 우려하셨던 **`_robust_json_parse` 파싱 에러가 아닌, 엔트로픽 API의 '한도 초과(Rate Limit)'로 인한 통신 차단**일 확률이 매우 높습니다.

원인 진단과 실질적인 해법을 정리해 드립니다.

### 1. 문제 원인 진단 (왜 Phase 2만 실패하는가?)

엔트로픽 콘솔 로그와 약 1분이라는 소요 시간은 API 통신 제한 시 나타나는 전형적인 패턴입니다.

- **토큰 없는() 항목의 반복:** API 요청이 서버에 도달했으나, 텍스트 생성을 시작하기 전에 거절(Reject)당했음을 의미합니다. 파이썬의 Anthropic SDK는 429(Rate Limit)나 529(Overloaded) 에러를 만나면 기본적으로 2회 자동 재시도(Auto-Retry)를 수행합니다. 로그에 항목이 주욱 이어져 찍힌 것은 이 재시도의 흔적입니다.
- **1분의 딜레이:** 재시도를 할 때는 지수 백오프(점점 대기 시간을 늘려가는 방식)가 적용됩니다. 재시도를 거듭하다 최종 실패하여 파이썬 코드로 예외(Exception)를 던지기까지 걸리는 시간이 통상 40초에서 1분 내외입니다.
- **Phase 2 병목:** Phase 1(Sonnet 4.5)은 토큰을 성공적으로 소모했지만, 직후에 실행되는 Phase 2(Sonnet 4.6)는 '기사 원문 + 식별된 패턴 + 윤리 규범 원문'까지 포함된 거대한 프롬프트를 전송합니다. 4건의 기사를 짧은 시간 안에 연달아 테스트하면서 엔트로픽의 **분당 토큰 제한(TPM, Token Per Minute)**을 순식간에 초과했을 가능성이 큽니다.

### 2. `_robust_json_parse` 이슈와의 관계

이전에 인지하셨던 `_robust_json_parse` 오류는 API가 텍스트를 성공적으로 모두 생성한 후(대시보드에 **출력 토큰이 정상적으로 찍힌 후**) 파이썬 단에서 JSON을 읽다가 실패할 때 발생합니다.

하지만 이번처럼 **로그에 토큰이 아예 기록되지 않았다면 파싱 단계까지 도달조차 하지 못한 것**입니다. 파이프라인의 `try-except` 블록이 API 통신 에러(`RateLimitError`)를 낚아채서 안전망으로 설계해둔 `"리포트 생성 중 오류가 발생했습니다..."` 메시지를 DB에 덮어씌운 상황입니다.

### ① [운영 레벨] 엔트로픽 콘솔 한도(Limits) 확인 및 상향

가장 근본적인 해결책입니다. 엔트로픽 콘솔의 **Settings > Limits** 메뉴에 들어가서 현재 계정의 요금제(Tier)와 TPM/RPM 한도를 확인해 보세요. 만약 Free Tier나 Build Tier 1에 머물러 있다면, Tier 2 이상으로 크레딧을 추가 결제하여 한도를 열어주는 것만으로 즉시 해결됩니다.

### ③ [코드 레벨] `_robust_json_parse` 3단계 폴백 적용 (미수정 이슈 대응)

이번 장애의 직접적 원인은 아니었으나, 언제든 발현될 수 있는 위험 요소입니다. 기존 함수를 아래와 같은 3단계 방어 로직으로 고도화하는 것을 권장합니다.

1. **1단계 (기존):** 정규식으로 마크다운 블록(```json)을 제거하고 표준 `json.loads` 시도
2. **2단계 (클리닝):** 후행 쉼표(Trailing commas)나 제어 문자 등 흔한 LLM JSON 문법 오류를 정규식으로 강제 교정 후 재시도
3. **3단계 (LLM 자가 교정):** 파싱 불가능 시, 가장 저렴하고 빠른 모델(Haiku)에게 깨진 문자열을 던져주어 "유효한 JSON으로 고쳐서 반환해"라고 1회 요청 (또는 최소한의 빈 딕셔너리 포맷 반환)
