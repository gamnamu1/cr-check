## 1. 프로젝트 구상 — 왜 만들었는가

CR-Check의 씨앗은 한국 언론 현실에 대한 한 가지 날카로운 질문에서 싹텄다. 언론윤리위반을 지적할 때 우리는 어떤 기준을 어떻게 적용하고 있는가? 직관과 관행에 의존하는 비판은 공정하지 않다. 반면, 기자들이 스스로 구독하고 있는 윤리규범을 준거점으로 삼아 분석한다면 이야기가 달라진다.

CR-Check는 이 원칙에서 출발했다. 한국 언론중재위원회의 심의 데이터 2,597건을 분석해 추출한 **119개의 문제적 보도관행 패턴**과, 기자들이 실제로 서명한 **14개 언론윤리규범 문서**를 결합하여, AI가 기사 하나를 분석하고 "이 기사는 어떤 패턴에서, 어떤 규범을 위반했는가"를 서술형으로 진단하는 공익 도구를 만들겠다는 것이었다.

점수를 매기지 않는다는 원칙은 처음부터 확고했다. "이 기사는 60점"이라는 판결이 아니라, "이런 맥락에서 이런 패턴이 보입니다. 해당 규범은 이렇게 규정하고 있습니다"라는 서술. 독자를 무력한 피해자가 아니라 스스로 판단하는 주체로 대우하는 방식이다.

기술적으로 가장 근본적인 문제의식은 **정적 매핑의 한계**였다. 기존 방식은 패턴과 윤리규범의 대응 관계를 JSON으로 하드코딩했다. 이 구조에서는 AI가 규범 원문을 인용할 때 환각이 발생하고, 새 패턴을 추가할 때마다 코드를 수정해야 하며, 매핑 근거를 설명할 방법이 없다. 이를 Supabase 단일 플랫폼의 **Hybrid RAG — Vector Search + Relational Graph**로 대체하는 것이 DB+RAG 마스터 플랜의 핵심이었다.

## 2. 기반 구축 — M0~M6, Phase D~F

### 2.1 데이터 설계 (M0~M2)

먼저 지식베이스를 쌓았다. 추가 규범 6개 원문을 수집하여 전체 14개 윤리규범 문서를 완성했고(2026-03-19), ethics_codes의 parent-child 매핑 394개 코드를 교차검증 에러 0건으로 완성했다(2026-03-23). 동시에 **골든 데이터셋** 26건을 구축했다. 실제 기사를 TP(True Positive — 문제 있는 기사) 20건과 TN(True Negative — 양질의 기사) 6건으로 분류하고, 각 기사에 대해 "반드시 탐지되어야 할 패턴 코드"를 레이블링한 이 데이터셋은 이후 모든 성능 측정의 기준이 된다.

데이터 누수(Data Leakage) 점검도 이 단계에서 완료했다. 26건 전부 CLEAN 판정을 받았으며, 포털 전재본 방식으로 저작권 원칙도 준수했다. 5AI 앙상블 교차검증으로 최종 검수까지 거쳤다.

### 2.2 임베딩·벤치마크 (M3)

M3는 첫 번째 냉혹한 현실 점검이었다. text-embedding-3-small 모델로 임베딩을 생성하고 골든셋을 돌렸다. 결과는 **FAIL**이었다. Recall@10(threshold=0.5, full article)이 **30.77%**. 정답 패턴을 상위 10개 후보 안에 담아내는 비율이 30%에 불과했다.

대분류별로 파고들면 더 처참했다. 1-1(진실성), 1-5(인권), 1-7(언어), 1-8(디지털) — 핵심 4개 대분류에서 Recall이 **0.0%**였다. 기사 본문과 패턴 설명문 사이의 의미적 거리를 벡터 검색이 전혀 건너지 못하고 있었다. 이것이 훗날 "38개 압축이 근본 병목"이라는 진단의 최초 신호였지만, 당시에는 아직 원인을 파악하지 못했다.

### 2.3 파이프라인 구현 (M4~M5)

M4에서 실제 파이프라인을 구현했다. 당시 설계는 2-Call 구조였다 — Haiku가 패턴 후보를 1차 확정하고, Sonnet이 최종 리포트를 생성하는 방식. M5 벤치마크에서 이 구조를 실측했다. Candidate Recall 50.8%, Final Recall **36.7%**, Precision **30.4%** — 목표치에 한참 미달했다.

특히 Haiku Suspect Accuracy가 **0.0%**이라는 수치가 뼈아팠다. Haiku에게 "의심 패턴"을 추리도록 했더니 아무것도 맞추지 못한 것이다. Haiku 단독 패턴 판단의 정밀도 한계가 분명해졌고, 이것이 이후 Sonnet Solo 구조로 전환하는 직접적 계기가 된다.

M5.5 종합 감리 과정에서 리포트 품질도 점검했다. 당시 설계대로라면 Sonnet이 `<cite ref="JCE-3"/>` 같은 태그만 출력하고 백엔드 CitationResolver가 DB 원문으로 치환하는 방식이었는데, 구현 과정에서 **Bug A(임베딩 벡터 미삽입)**, **Bug B(JSON 파싱 전체 실패)**, **Bug C(규범 조회 간헐적 0건)** 세 독립 버그가 발견되었다. 앙상블 감리(Claude + Manus + Gemini + Perplexity) 교차검증으로 원인을 확정하고 수정했다. 이 과정에서 CitationResolver는 결국 비활성화되었고, 규범 원문을 `〔〕` 마커로 Sonnet이 직접 출력하는 방식으로 전환했다.

### 2.4 배포 및 아카이빙 (Phase D~F)

Phase D는 분석 결과의 영속성과 공유 가능성을 만드는 단계였다. Supabase에 `analysis_results` 테이블을 설계하고, 12자리 `share_id`를 발급하여 `/report/{share_id}` URL로 누구나 결과를 볼 수 있게 했다. URL 정규화(UTM 파라미터 제거), 캐시 시스템, Supabase heartbeat GitHub Actions 크론잡까지 꼼꼼히 구현했다. 4인 감리(Antigravity, Manus, Gemini, Perplexity) 피드백 6건을 반영했다.

Phase E는 클라우드 배포였다. Railway(백엔드) + Vercel(프론트엔드) + Supabase(DB) 삼각 구조로 배포했다. 배포 후 TP 기사 분석이 7건 중 4건 실패하는 위기가 왔다. `_robust_json_parse` 4단계 폴백, 529 OverloadedError 백오프 강화, `inference_role` 컬럼 마이그레이션 — 세 버그를 각각 진단하고 수정했다(Phase 2 Bugfix).

Phase F는 예약된 테스트셋(Reserved Test Set) 73건을 격리 주입 방식으로 검증하는 단계였다. 2026-04-11 완료. TN 탐지 **100% 정확도**를 확인했다. 정량 지표(precision_macro 0.28, recall_macro 0.60)는 낮았지만, 이 단계의 진짜 가치는 "모델 튜닝이 아닌 데이터 레이어 보강이 1순위"라는 방향성을 공식 확인한 것이었다.

## 3. 성능 개선 트랙 — Phase G → H → STEP 0~6

### 3.1 Phase G: DB 및 RAG 구조 개선

Phase F 완료 후, 프로젝트는 성능 개선 트랙에 본격적으로 진입했다. Phase G는 두 가지 흐름으로 진행됐다.

첫 번째는 **매핑 복원 작업**이었다. `pattern_ethics_relations` 테이블을 정밀 검수했다. PRG-T1·T2 교정(STEP 0), Phase γ 삭제 7건 역추적(STEP 0.5), 공백 패턴 13개 중 11개 매핑 복원(STEP 1)을 순차적으로 완료했다. 결과적으로 relations 총 91건까지 확대됐다.

두 번째는 **성능 실험**이었다. R4에서 R7까지 네 가지 구성을 순차적으로 벤치마크했다.

| 버전 | 변경 내용 | FR | Precision | TN FP Rate |
|---|---|---|---|---|
| R4 | 기준선 (이진 게이트 있음) | 26.7% | 30.4% | 2/6 |
| R5 | 이진 게이트 제거, 독립 평가 | 36.7% | 44.2% | 6/6 |
| R6 | R5 + 1-1-4 가드 + 예시10 | 22.5% | 25.8% | 3/6 |
| R7 | R5 + 1-1-4 가드만 | 20.0% | 25.8% | 4/6 |

이 실험에서 두 가지 결정적 사실이 드러났다.

**첫째, 이진 게이트가 병목이었다.** Devil's Advocate CoT의 "(가) 양질 판정 → detections=[]" 구조는 벡터 검색이 정답을 찾아줬는데도 46%를 기각하고 있었다. 이 게이트를 제거하자(R5) FR이 26.7% → 36.7%로 10pp 상승했다.

**둘째, 프롬프트 패치는 효과가 없었다.** 오탐을 줄이려고 가드 규칙을 추가하자(R6, R7) FR이 오히려 R4보다 낮아졌다. 결론은 명확했다 — *패턴 판단 기준은 메인 프롬프트가 아니라 DB description 필드에 있어야 한다.* 이는 프롬프트 땜질을 중단하고 데이터 레이어로 내려가야 한다는 선언이었다.

Layer 2 교차 감리(Antigravity, Perplexity, Gemini, Manus 4인)를 통해 확정 소견 7개, 잠정 소견 8개를 도출하고 SQL 수정을 완료했다.

골든셋 TN 라벨도 이 시점에 재검토하여 확정했다. C-02(트랜스젠더 보고서)는 경계선 케이스로 제거, E-17(AI 여성 성희롱)은 TN → TP 전환(7-5 패턴 실재 확인), C2-07(대리 입영 적발)은 TN 유지. 결과적으로 TP 21건 + TN 5건 = 26건으로 확정됐다.

### 3.2 Phase H: 패턴 세분화 (38→107개 leaf)

R4~R7 실험이 가리킨 근본 원인으로 돌아갔다. **38개 소분류 description 안에 여러 독립 개념이 뭉쳐 있다.** 임베딩 벡터의 방향성이 분산되어 Sonnet이 패턴 간 경계를 모호하게 인식한다. 이것은 프롬프트로 해결할 수 없는, 데이터 구조의 구조적 문제다.

Phase H의 목표는 38개 소분류를 **107개 leaf 패턴**으로 세분화하는 것이었다. 동시에 `search_text` 필드를 `description`에서 분리하여, 벡터 검색용 간결한 어휘와 리포트 생성용 품격 있는 정의가 하나의 필드를 공유하면서 생기는 트레이드오프를 제거하기로 했다.

두 계획 문서(PHASE_H_EXECUTION_PLAN_v1.0.md + PIPELINE_IMPROVEMENT_PLAN_v1.1.md)를 실행 순서 기반으로 통합하여 **MASTER_EXECUTION_PLAN_v1.0.md**를 작성했다(2026-04-28). 이 문서가 이후 모든 작업의 단일 진실 원천이 되었다.

### 3.3 STEP 0~3: DB 마이그레이션

**STEP 0**에서는 패턴과 규범 DB 관련 판단 작업을 수행했다. 특히 STEP 0-B에서 `applicable_contexts` 값 9개를 확정했다 — NULL(범용), `{general}`, `{health}`, `{military}`, `{election}`, `{crisis}`, `{disaster}`, `{unification}`, `{crime}`. 5인 교차 감리(Antigravity, Gemini, Manus, NotebookLM, Perplexity) 완료.

**STEP 1**에서는 107개 leaf 코드 체계를 설계했다. 기존 38개 소분류가 107개 leaf로 세분화되고, 각 leaf는 대분류 8개 → 부모 34개 → leaf 107개의 3단 계층 안에 자리를 잡았다. 6인 교차 감리 완료.

**STEP 2**에서는 실제 DB 마이그레이션을 실행했다. `search_text` 컬럼 추가, `detection_strategy` 컬럼 추가(vector/structural 구분), `is_meta_pattern` 플래그, updated_at 트리거 등 신규 컬럼을 추가하고 107개 패턴을 INSERT했다. 모든 DML은 기획자가 SQL Editor에서 직접 실행했다.

**STEP 3**에서는 `pattern_ethics_relations` 테이블을 기존 38개 코드 체계에서 107개 leaf 체계로 재배분했다. 4단계 배치 작업을 거쳐 활성 leaf 76개 전원이 매핑되었으며, 최종 관계 건수는 111건 이상, PCP(Pattern Coverage Percentage) 44.6%를 달성했다. 2026-05-03 완료.

### 3.4 STEP 4~5: 코드 품질·프롬프트 개선

**STEP 4**는 세 개의 하위 작업으로 구성됐다.

STEP 4-A에서는 `_build_ethics_context()` 함수를 재작성했다. primary(violates, strong/moderate)와 reference(related_to, moderate) 두 그룹을 분리하고, weak와 exception_of 항목을 필터링하여 Sonnet이 받는 윤리 컨텍스트의 노이즈를 대폭 줄였다.

STEP 4-B에서는 `analysis_ethics_snapshot`을 재활성화했다. CitationResolver 비활성화 이후 멈춰 있던 스냅샷 저장 기능을 복구하여, 과거 분석 당시 인용된 규범 원문을 추적할 수 있게 했다.

STEP 4-C에서는 `get_ethics_for_patterns` RPC를 수정했다. `article_context` 파라미터를 추가하여 기사 유형에 따른 윤리규범 필터링이 가능해졌다 — 감염병 기사에서만 감염병보도준칙이 활성화되는 방식이다. 6인 감리 완료. 2026-05-06 완료.

**STEP 5**는 Sonnet 프롬프트의 전면 재설계였다.

STEP 5-A(2026-05-08 완료)에서는 `pattern_matcher.py`를 전면 재작성했다. 핵심은 세 가지였다. 첫째, `_load_pattern_catalog()`가 DB에서 107개 leaf 패턴을 동적으로 로드하도록 재구현했다. 둘째, `_build_pattern_list_text()`가 [섹션0] 공통 안내 / [섹션1] 벡터 검색 기반 패턴 / [섹션2] 구조적 판단 필수 패턴의 3섹션 구조를 출력하도록 했다. 셋째, 8개의 few-shot 예시를 구버전 코드에서 v3 leaf 코드로 전면 교체했다. 구조적 패턴(3-1-b 등)에는 matched_text가 아닌 구조 묘사 예시를 별도로 추가했다.

STEP 5-B에서는 혼동 쌍(Confusion Pairs) 5개를 확정하여 DB화하고, distinction_guide v4와 실행 패키지를 작성했다.

### 3.5 STEP 6: 임베딩 재생성

STEP 6(2026-05-10 완료)은 이 모든 작업의 수렴점이었다.

Phase A에서 사전 점검을 했다. active·vector·leaf 패턴 64건, search_text NULL/공백 0건을 확인했다. search_pattern_candidates RPC가 is_meta_pattern=FALSE 단독 필터만 갖고 있어 4-필터 마이그레이션이 필요함을 확인했다.

Phase B-7에서 RPC를 업그레이드했다. `is_active=TRUE`, `detection_strategy='vector'`, leaf 코드 정규식 패턴(`^[0-9]+-[0-9]+-[a-z]+$`), `description_embedding IS NOT NULL` — 4개 필터를 적용하여 벡터 검색이 오직 64개 활성 vector leaf 패턴에서만 동작하도록 격리했다.

Phase C에서 `generate_embeddings.py`를 8개 항목 변경했다. `--patterns-only`, `--dry-run` 플래그를 추가하고, fetch_patterns()에 4-필터 SQL을 적용하고, `search_text` 단독 입력으로 임베딩을 준비했다.

## 6. 워크플로우 특징 — 삼각편대 감리 체계

CR-Check 프로젝트의 가장 독특한 특징 중 하나는 **삼각편대 워크플로우**다.

**Claude.ai (설계·감독)**: 전체 방향을 설계하고, CLI에 전달할 단계별 프롬프트를 작성하며, 감리 의견을 취합하고 합의 판단을 제공한다.

**Claude Code CLI (실행)**: 코드 수정, 파일 생성, SQL 초안 작성 등 실행 작업을 맡는다. 그러나 DB DML(INSERT, UPDATE)은 절대 자동으로 실행하지 않는다.

**기획자 감나무 (최종 결정권자)**: 모든 DB 변경은 기획자가 Supabase SQL Editor에서 직접 실행한다. 이는 단순한 보안 원칙이 아니라, 데이터에 대한 기획자의 직접적 책임과 이해를 보장하는 철학적 선택이다.

외부 감리자들(Antigravity, Perplexity, Gemini, Manus, NotebookLM)은 독립적인 관점을 제공한다. 2026-05-02 이후 확정된 원칙에 따라, 모든 감리자는 역할을 분리하지 않고 **동일한 프롬프트로 중복·교차 검증**한다. 이렇게 얻은 의견에서 가중치를 확인하고, 예외 소견을 식별하여 최종 판단에 반영한다.

SQL 자동 실행 금지, diff 선제시 → 승인 → 커밋 순서, STEP 단위 승인 게이트, ON CONFLICT DO NOTHING 원칙 — 이 절대 원칙들은 속도를 희생하고 안전성을 택하는 것이 아니라, 각 변경의 의도를 명확히 하고 롤백 가능성을 확보하는 엔지니어링 철학의 표현이다.
